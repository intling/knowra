# knowra 前端章程

## 1. 端侧定位

**knowra 前端** 是 knowra 的用户界面层，负责承载知识库管理、资料接入、问答交互、引用展示和系统状态反馈等用户可见体验。

前端不单独定义产品范围和业务优先级。产品目标、跨端边界和 OpenSpec 治理以根目录 `AGENTS.md` 为准；具体功能范围、验收标准和阶段任务以 OpenSpec 变更文档为准。

## 2. 技术栈基线

- **框架**：Vue 3.5.0
- **语言**：TypeScript 5.9.0
- **构建工具**：Vite 8.0.12
- **路由**：Vue Router 4.5.0
- **状态管理**：Pinia 3.0.0
- **样式**：Tailwind CSS 4.0.0
- **测试**：Vitest 4.0.0 + `@vue/test-utils` 2.4.0
- **代码规范**：ESLint 9.0.0 + Prettier 3.0.0
- **运行环境**：Node.js 24.15.0，npm 11+

依赖版本以 `package.json` 和 lockfile 为准。升级框架、构建工具或测试工具时，应通过 OpenSpec 说明动机、影响范围和验证方式。

## 3. 前端职责

前端主要负责：

- 页面结构、路由组织和应用壳层。
- 文档上传、知识库浏览、问答输入、回答展示和引用来源展示等交互界面。
- 前端状态管理，包括加载、错误、筛选、会话态和轻量 UI 状态。
- API client 封装和请求错误展示。
- 与后端健康检查、文档、检索、问答等接口的联调。
- 基础可访问性、响应式布局和用户反馈。

前端不负责数据库访问、文档解析、向量化、检索算法、模型调用或 RAG 编排。这些能力应通过后端 API 暴露。

## 4. 推荐目录边界

前端代码应按职责拆分，避免把业务逻辑集中在单个组件中。

```text
front/src/
├── api/          # API client 与接口函数
├── assets/       # 静态资源
├── components/   # 可复用展示与交互组件
├── layouts/      # 页面布局
├── router/       # 路由定义
├── stores/       # Pinia store
├── views/        # 页面级组件
├── App.vue       # 应用壳层
├── main.ts       # 应用入口
└── style.css     # 全局样式入口
```

新增目录前应先确认它代表稳定职责边界，而不是临时分类。

## 5. API 与配置规范

- 所有后端请求统一使用 `/api` 前缀。
- 开发环境通过 Vite Proxy 转发到后端服务，业务代码不得写死 `localhost:8000`。
- API 基础路径通过 `VITE_API_BASE_URL` 配置，默认值为 `/api`。
- API 类型应尽量在接口层集中定义或复用，避免在多个组件中重复手写响应结构。
- 请求错误应在 API 层保留可诊断信息，在页面层转化为用户可理解的反馈。

## 6. UI 与交互规范

- 页面应服务于知识库和问答工作流，优先保持清晰、安静、可扫描。
- 常用操作应有明确反馈，包括加载中、空状态、错误状态和成功状态。
- 引用来源、文档片段、检索结果等信息应便于用户验证，不应只展示模型回答文本。
- 组件应围绕真实复用场景抽象，避免过早建立复杂组件库。
- 样式优先使用 Tailwind CSS 和项目内已有样式约定。

## 7. 质量门禁

前端变更完成前，应根据影响范围运行必要验证命令：

```bash
npm run lint
npm run test
npm run build
```

涉及页面交互、布局或用户流程的改动，应进行本地浏览器验证。涉及跨端接口的改动，应同时确认后端接口契约和代理配置。

## 8. 文档同步

以下变化需要同步更新文档：

- 启动方式、环境变量、端口或代理配置变化。
- 前端目录结构或技术栈变化。
- 跨端 API 调用方式变化。
- 用户可见工作流、页面入口或核心交互变化。

同步目标通常包括 `front/README.md`、根目录 `README.md`、相关 OpenSpec artifacts，以及必要时更新本章程。
