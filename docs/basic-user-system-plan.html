knowra 基础用户体系方案

本方案用于探索 knowra 在暂不实现登录、注册、鉴权的前提下，如何先建立可关联业务数据的用户基础模型， 为后续文件上传、知识库归属、团队协作、登录注册与权限扩展保留稳定边界。

日期：2026-05-23 阶段：探索方案 范围：基础用户数据支撑 不含：登录、注册、会话、鉴权

1. 目标与边界

本阶段要解决
  • 系统中存在稳定的 User 数据实体。
  • 上传文件、文档、知识库等业务表可以通过 user_id 关联用户。
  • 提供开发阶段可用的默认用户或内部用户解析方式。
  • 后续接入登录注册时，不需要大规模重写业务表外键。
本阶段不解决
  • 不提供用户注册接口。
  • 不提供密码登录、OAuth、短信、邮箱验证码等认证能力。
  • 不引入访问令牌、Cookie Session 或权限中间件。
  • 不实现多租户团队权限模型。
核心判断：现在需要的是“用户归属数据契约”，不是“认证系统”。先把业务数据和用户标识的关系固定下来，后续认证只负责把请求映射到已有用户。

3. 推荐领域模型

本阶段建议只建一个 users 表，字段保持小而稳定，避免提前设计复杂认证表。

字段 类型建议 是否必需 说明
id UUID 是 用户主键。推荐 UUID，避免后续多环境合并或外部身份映射时依赖自增序列。
display_name varchar(100) 是 展示名称。本阶段可默认 Default User。
email varchar(255) 否 为后续邮箱登录预留。允许为空，非空时唯一。
avatar_url varchar(1024) 否 为后续用户资料展示预留。
status varchar(32) 是 建议初始支持 active、disabled，默认 active。
deleted_at timestamptz 否 软删除时间。为空表示用户未删除；非空表示用户信息已删除，业务查询默认排除。
created_at timestamptz 是 创建时间。
updated_at timestamptz 是 更新时间。
不建议现在加入 password_hash、last_login_at、email_verified_at、oauth_provider 等认证字段。这些字段属于认证能力，不属于基础用户归属模型。

软删除语义

  • 用户删除时只写入 deleted_at，不物理删除 users 记录。
  • 常规用户查询、当前用户解析和资源归属检查默认只返回 deleted_at IS NULL 的用户。
  • 已删除用户关联过的文件、文档、知识库等历史资源仍能保留可追溯的 owner_user_id。
  • status 用于表达禁用、停用等可恢复状态；deleted_at 用于表达用户信息已被删除，两者不要混用。

外键命名建议

对资源归属类字段，优先使用 owner_user_id，语义比 user_id 更清楚。 如果某个表记录的是操作者行为，例如审计日志、任务触发记录，可以使用 created_by_user_id 或 actor_user_id。

字段名 适用场景
owner_user_id 文件、文档、知识库、集合等资源归属。
created_by_user_id 记录谁创建了某条业务记录。
updated_by_user_id 需要记录最后修改者时使用。
actor_user_id 事件、任务、审计日志中的行为主体。

5. 当前用户解析策略

因为本阶段没有登录态，仍然需要一个稳定方式让业务代码拿到“当前用户”。推荐分两层处理。

开发期默认用户

  • 在数据库中维护一个默认用户，例如 Default User。
  • 后端提供 get_current_user() 依赖或服务函数，当前总是返回这个默认用户。
  • 文件上传、文档创建等业务统一通过该函数获取用户，而不是在各处硬编码默认 UUID。

可选请求头覆盖

  • 开发阶段可支持内部请求头 X-Knowra-User-Id，便于测试不同用户的数据隔离。
  • 该能力必须明确标注为开发/内部用途，生产前需要被真实认证机制替代或关闭。
  • 如果请求头用户不存在，应返回清晰错误，而不是自动创建未知用户。
推荐路径：先实现默认用户解析；等文件上传或知识库功能需要多用户测试时，再加 X-Knowra-User-Id 开发覆盖能力。

6. API 边界建议

本阶段 API 不应变成用户管理后台。建议保持非常薄。

接口 阶段 用途
GET /api/users/me 建议纳入 返回当前解析出的用户，用于前端、上传流程、调试和联调。
GET /api/users/{user_id} 可暂缓 如果没有管理场景，先不暴露单用户查询。
POST /api/users 不建议本阶段开放 容易与注册语义混淆。内部初始化用服务函数、迁移或种子脚本处理。
PATCH /api/users/me 后续按需 用户资料编辑需要前端场景后再加。

7. 前端接入范围

前端本阶段只消费“当前用户”数据，不承担注册、登录、鉴权或资源归属判定。资源归属应由后端根据当前用户解析结果写入， 前端不应在上传文件等普通用户流程中手动传入 owner_user_id。

位置 建议内容 说明
front/src/api/users.ts 定义 User 类型与 getCurrentUser()。 调用 GET /api/users/me，复用现有 apiGet，不要在业务代码中拼接后端地址。
front/src/stores/user.ts 维护 currentUser、isLoading、error 和 loadCurrentUser()。 用户是跨文件上传、知识库、问答等功能都会依赖的基础状态，建议独立于 app store。
front/src/App.vue 或后续 layout 应用启动时触发当前用户加载，并在顶部或侧边栏展示用户名称。 只展示身份上下文，例如 Default User；不展示登录、退出、注册入口。
front/src/views/HomeView.vue 可在开发期展示当前用户加载结果和错误状态。 用于联调用户基础能力；后续有正式布局后可迁移到应用壳层。
front/src/api/client.ts 可预留开发期请求头注入点。 如后端启用 X-Knowra-User-Id 测试多用户隔离，前端应只通过开发配置或测试工具使用，不做成普通用户 UI。

前端用户类型建议

export interface User {
  id: string
  displayName: string
  email: string | null
  avatarUrl: string | null
  status: "active" | "disabled"
  deletedAt: string | null
  createdAt: string
  updatedAt: string
}

如果后端响应使用 snake_case，前端可以先保持接口类型与响应一致，例如 display_name、deleted_at。 是否在 API 层转换成 camelCase，应在 OpenSpec 中统一约定，避免不同接口风格混用。

前端交互边界

  • 不新增登录、注册、退出、找回密码等页面或按钮。
  • 不提供用户选择器给普通用户切换身份。
  • 上传文件、创建知识库等操作不让用户填写 owner_user_id。
  • 如果 GET /api/users/me 失败，页面应显示当前用户不可用的错误状态，并阻止依赖用户归属的操作。
  • 如果后端因默认用户被软删除而返回错误，前端应展示可诊断反馈，而不是静默降级为匿名用户。

8. 后续认证扩展路径

后续登录注册可以在不破坏 users.id 和业务外键的前提下扩展。

当前阶段
┌──────────────┐
│    users     │
└──────┬───────┘
       │
       ▼
业务资源通过 owner_user_id 关联用户

后续认证阶段
┌──────────────┐       ┌─────────────────┐
│    users     │◀──────│ user_identities │
└──────┬───────┘       │ provider        │
       │               │ provider_uid    │
       ▼               │ password_hash   │
业务资源不需要变更       └─────────────────┘
未来能力 建议扩展方式
邮箱密码登录 新增 user_credentials 或 user_identities 表保存密码哈希，不把密码字段混入基础 users 表。
OAuth 登录 通过 provider、provider_user_id 映射到 users.id。
邮箱验证 新增验证令牌或验证状态字段，避免现在提前污染模型。
团队/组织 新增 organizations、organization_members，资源归属再按场景扩展为用户归属或组织归属。
权限控制 在认证之后引入角色、权限或资源授权表，当前阶段只保证数据可追溯到用户。

9. OpenSpec 变更建议

如果进入实施，建议创建一个独立 OpenSpec 变更，例如 add-basic-user-foundation。

建议范围

  • 新增 users 数据模型与迁移。
  • 新增当前用户解析服务，默认返回系统默认用户。
  • 新增 GET /api/users/me。
  • 用户表支持 deleted_at 软删除字段，默认查询排除已删除用户。
  • 前端新增当前用户 API 封装和轻量 user store。
  • 前端展示当前用户上下文和用户加载失败状态，但不新增登录注册 UI。
  • 为未来上传文件等功能规定资源表应通过 owner_user_id 关联用户。

验收标准

  • 迁移执行后数据库存在 users 表。
  • 系统可获得一个默认 active 用户。
  • GET /api/users/me 返回 200 和稳定用户结构。
  • 软删除用户不会被当前用户解析服务返回。
  • 前端可以通过 getCurrentUser() 获取当前用户。
  • 前端 user store 能表达加载中、加载成功和加载失败三种状态。
  • 前端没有登录、注册或用户切换入口。
  • 服务层不在业务代码中硬编码默认用户 ID。
  • 后续业务表可通过外键引用 users.id。

10. TDD 实施顺序

项目章程要求实现类任务遵循 Red-Green-Refactor。后续真正编码时建议按以下小循环推进。

  1. 先写 GET /api/users/me API 测试，确认当前失败，失败原因是路由不存在。
  2. 写最小路由和 schema，让接口结构通过，但用户查询可先由服务返回固定测试对象。
  3. 写用户服务测试，要求能够获取默认用户，确认失败。
  4. 实现 User 模型、服务查询逻辑和测试数据库准备。
  5. 写软删除查询测试，要求 deleted_at 非空的用户不会被当前用户解析服务返回。
  6. 写迁移验证或模型元数据测试，确保 users 表字段、唯一约束和 deleted_at 字段符合方案。
  7. 重构路由、服务和 schema 边界，保持测试通过。
  8. 运行 uv run ruff check .、uv run ruff format --check .、uv run pytest。
  9. 写前端 getCurrentUser() API 测试，确认当前失败，失败原因是封装不存在。
  10. 实现最小 front/src/api/users.ts，让 API 测试通过。
  11. 写 user store 测试，覆盖加载中、成功写入当前用户、失败写入错误状态。
  12. 实现最小 user store，并在页面或应用壳层消费该状态。
  13. 运行 npm run lint、npm run test、npm run build。

11. 风险与取舍

风险 影响 处理方式
默认用户被业务代码硬编码 后续接入真实登录时改动面扩大 所有业务通过 get_current_user() 或用户服务获取用户。
过早加入认证字段 模型职责混乱，后续 OAuth/密码登录难以统一 认证信息后续独立成表，users 只保存用户资料与状态。
API 暴露创建用户 容易形成事实上的注册接口 本阶段不开放 POST /api/users，默认用户通过内部初始化保证。
资源表外键命名不统一 上传、文档、知识库等模型难以形成一致约定 资源归属统一使用 owner_user_id。
前端直接提交 owner_user_id 资源归属可能被客户端伪造，后续接入认证时需要重写流程 普通业务操作由后端根据当前用户解析结果写入归属，前端只消费当前用户上下文。
前端提前设计登录注册 UI 用户预期被提前锁定，后续认证方案调整成本上升 本阶段只展示当前用户状态，不提供认证入口。
需要在 OpenSpec 中明确“无登录注册”的阶段边界，否则实现时很容易把用户基础模型扩展成半套认证系统。

12. 推荐结论

先实现一个极简但稳定的 users 表、当前用户解析服务和 GET /api/users/me。 前端只增加当前用户 API 封装、轻量 user store 和必要状态展示。所有后续资源类能力从一开始通过后端解析的当前用户写入 owner_user_id。 认证、注册、登录、权限和组织协作都作为后续独立 OpenSpec 变更扩展。

这个方案可以满足当前“上传文件可关联用户”的基础需求，同时避免把尚未确定的认证形态提前写死在核心数据模型里。
