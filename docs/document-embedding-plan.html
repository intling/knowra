knowra 文档向量化方案（Document Embedding）

本文沉淀"在现有文档分块能力之后，新增基于 pgvector 的文档块向量化与向量存储能力"的方案。范围覆盖 Embedding 模型选型、向量化作业模型、Embedding 适配器、向量化 API、与分块结果的关系设计，以及后续语义检索的扩展边界（pgvector 索引与查询策略留待后续阶段）；不在本文中实现代码。

阶段：方案探索 日期：2026-06-23 推荐变更：add-document-embedding 向量引擎：pgvector 推荐模型：BGE-M3 (1024d) 部署方式：Ollama 本地服务 / sentence-transformers 直连 分块自动触发向量化：内存直传 chunk text 前置能力：file-upload-storage ✓、document-parsing ✓、document-chunking ✓

1. 方案结论

推荐把"文档向量化"设计为分块流程的内嵌步骤：分块作业成功后，在同一后台任务中直接将分块结果文本（contextualized_text）传入 Embedding 适配器生成向量，写入带 pgvector 类型的 document_chunk_embeddings 表。首版使用与解析/分块相同的 BackgroundTasks 调度器，分块成功后自动在同一任务上下文中执行向量化。重新分块时自动触发重新向量化；同时提供独立的 /reembed API 支持仅更换 Embedding 模型而不重新分块。

向量化的核心价值是：将不可直接计算的文本块转化为高维语义向量，使系统能够进行语义相似度比较——这是语义检索和 RAG 问答的前提能力。不是简单地把文本变成数字，而是用预训练语言模型将文本映射到一个语义空间，在这个空间里语义相近的文本向量距离也相近。

首个版本的成功标准不是"向量化后立刻可以语义检索和 RAG 问答"，而是"每个已分块文档的 chunks 被可靠地转化为可供后续检索消费的向量，向量持久化存储并可通过 pgvector 算子查询"。

向量化触发策略：分块作业成功后在同一后台任务中自动执行向量化，chunk 文本（contextualized_text）直接从已持久化的 chunk 记录读取（文本已在内存或可通过 storage_key 读取）。前置条件：只有分块作业成功（status=succeeded）时才允许触发向量化；分块失败或分块被 superseded 的作业不应触发向量化。独立向量化 API：提供 POST /reembed API 支持在分块结果不变的情况下，更换 Embedding 模型或参数重新生成向量，旧向量作业标记为 superseded。重新分块时自动重新向量化：当用户通过 /rechunk 触发重新分块时，在新的分块作业成功后自动执行向量化，旧向量化作业跟随旧分块作业被标记为 superseded。存储策略：向量本身始终存储在 pgvector 列中，不走文件存储（向量维度固定，存储开销可预测）。向量化配置快照（模型名称、维度、batch_size 等）写入作业表的 embed_config_json 供审计。

2. 现有基础与约束

项目现状 对向量化方案的影响
已有 document_chunks 表，包含 text、contextualized_text、token_count、heading_path、page_numbers 等字段；分块文本 ≤2KB 直接入库、>2KB 走文件存储。 向量化输入使用 contextualized_text（带层级标题上下文的富文本）。需处理长文本从文件读取的情况。Chunk 的其他元数据（heading_path、page_numbers）不需要向量化但会在检索结果中一并返回。
已有 document_chunk_jobs 表，分块作业存在 superseded 状态；查询 chunks 时仅返回活跃作业的结果。 向量化作业沿用相同的 superseded 机制：当向量化作业被新作业取代时标记为 superseded，查询时仅返回活跃向量化作业的结果。重新分块时，新分块作业的向量化自动取代旧向量化。
分块 superseded 后旧 chunk 数据保留不删除，查询只返回最新活跃作业的 chunk。 向量化结果同样保留不删除；查询时通过活跃 embedding_job_id 过滤。旧向量仍占用存储空间，后续可提供清理 API。
项目已安装 pgvector>=0.4.0（Python 客户端）；PostgreSQL 已启用 vector 扩展（最初 migration 已执行 CREATE EXTENSION vector）。 可直接使用 pgvector 的 vector(N) 类型。后续语义检索阶段再评估合适的索引策略。Alembic migration 需新增向量列，须使用原始 SQL（SQLModel 不原生支持 pgvector 类型）。
解析和分块在 BackgroundTasks 内同步执行（run_parse_job 函数）。 向量化沿用同一模式：在分块成功后、run_parse_job 结束前同步调用向量化服务。首版不引入独立 worker 或队列。
外部能力由 Adapter 封装（DoclingParserAdapter、DoclingChunkerAdapter），SDK 类型不泄露到服务层和路由层。 向量化遵循同一模式：EmbeddingAdapter 封装具体的 Embedding 模型调用（Ollama API / sentence-transformers / 云 API），返回项目内的规范化向量结果。不把第三方 SDK 类型扩散到外部。
项目配置使用 Pydantic Settings + .env 管理。 向量化配置项（Embedding 模型名称、维度、服务地址、batch_size 等）加入 Settings 类。通过 EMBEDDING_DISPATCHER 预留后续切换调度器的空间。
当前用户模型使用硬编码 DEFAULT_USER_ID，无认证系统。 向量化结果沿用 owner_user_id 归属，所有查询和操作需校验用户权限。

3. Embedding 模型选型

选型需同时满足：(1) 中英双语检索质量好，(2) 可本地部署保护隐私（与 knowra 的产品定位一致），(3) 硬件要求适中（不强制 GPU），(4) 生态成熟、接入成本低。

候选模型对比

✅ BGE-M3 (BAAI/bge-m3) — 推荐

维度：1024

特点：多语言（中英均强）、支持 Dense + Sparse 混合检索、1024 维在质量和存储间平衡好、MTEB/C-MTEB 排名靠前。

部署：Ollama 一键拉取（ollama pull bge-m3），~2.2GB 量化版本；也可通过 sentence-transformers 直接加载。

优点：BGE 系列中文社区使用最广泛、资料丰富；Ollama 部署极简；1024 维度适中，不会过度膨胀索引。

缺点：量化版精度略低于全精度；Ollama 服务需单独启动。

⭐ Qwen3-Embedding (0.6B) — 备选

维度：4096

特点：2025 年 Qwen3 系列新模型，多语言 MTEB 排名顶尖，支持 Reranking。

部署：Ollama 可拉取（ollama pull qwen3-embedding），~1.2GB 量化版（0.6B 参数版）。

优点：检索质量最优；0.6B 小模型 CPU 可运行。

缺点：4096 高维度增加索引和存储开销（约 BGE-M3 的 4 倍）；社区生态较 BGE 新。

text2vec-large-chinese

维度：1024

特点：纯中文模型，中文语义理解好。

缺点：英文能力弱，不适合中英混合文档。

OpenAI text-embedding-3-small

维度：512（可调至 1536）

特点：质量好、API 稳定。

缺点：数据发送到外部（与 knowra 隐私定位冲突）；按 token 计费、大文档库成本高；依赖网络。

推荐 BGE-M3 作为首版 Embedding 模型。理由：(1) 多语言能力覆盖中英文文档；(2) 1024 维在检索质量和存储/索引开销间取得平衡；(3) Ollama 部署极简（ollama pull bge-m3 && ollama serve），适合个人知识库场景；(4) BGE 系列是中文 RAG 社区的事实标准，社区支持丰富；(5) 如果后续需要更高检索质量，可切换到 Qwen3-Embedding（通过 /reembed 触发重新向量化，无需重新分块）。

Embedding 服务部署方式

✅ 首版：Ollama 本地服务

原理：Ollama 作为独立的 HTTP 服务运行，提供兼容 OpenAI 的 /api/embeddings 端点。后端通过 HTTP 调用生成向量。

优点：模型管理由 Ollama 负责（下载、量化、缓存）；服务解耦，重启后端不影响已加载模型；GPU 加速自动启用（如有 GPU）；CPU 亦可运行。

缺点：需要额外启动 Ollama 服务；首次拉取模型需下载。

配置：embedding_service_url=http://localhost:11434

备选：sentence-transformers 直连

原理：后端进程中直接加载 HuggingFace 模型（通过 sentence-transformers 库），调用 model.encode() 生成向量。

优点：无外部服务依赖，部署最简单；延迟更低（无网络开销）。

缺点：模型加载到后端进程内存中，增加内存占用；Python 进程启动变慢；不支持 GPU 时 CPU 推理较慢。

配置：embedding_dispatcher=inline

采用 Adapter 模式：定义 EmbeddingAdapter 抽象接口，首版提供 OllamaEmbeddingAdapter 实现。适配器接口仅暴露 embed(texts: list[str]) → list[list[float]]，调用方不感知底层是 Ollama 还是其他服务。后续可随时新增 SentenceTransformersEmbeddingAdapter 或 OpenAIEmbeddingAdapter，通过配置切换。

4. 推荐数据流

1. 分块完成 chunk 文本（contextualized_text）已持久化，chunk_job 状态为 succeeded。
2. 创建向量化作业 分块成功后自动创建，写入 document_embedding_jobs。
3. 读取 chunk 文本 从 document_chunks 读取每个 chunk 的 contextualized_text（数据库或文件）。
4. 批量生成向量 调用 Ollama /api/embeddings，batch 发送 chunk texts，返回 embedding vectors。
5. 保存向量 向量写入 document_chunk_embeddings 表的 vector 列，关联 chunk_id 和 job_id。 
run_parse_job 执行流程（扩展现有逻辑）：
        |
        +--> 解析原始文件 → DoclingDocument
        |
        +--> 持久化 parsed_documents + document_segments
        |
        +--> [document_chunking_enabled?]
        |       |
        |       +--> DoclingChunkerAdapter.chunk(docling_document)
        |       |       |
        |       |       +--> document_chunk_jobs (succeeded)
        |       |       |
        |       |       +--> document_chunks (N 条)
        |       |
        |       +--> [document_embedding_enabled?]  ← 新增加
        |               |
        |               +--> 创建 document_embedding_jobs (running)
        |               |
        |               +--> 从 document_chunks 读取 contextualized_text
        |               |
        |               +--> EmbeddingAdapter.embed(texts) → vectors
        |               |
        |               +--> 逐条写入 document_chunk_embeddings
        |               |
        |               +--> document_embedding_jobs (succeeded)

重新分块 (POST /rechunk)：
  1. 重新解析原始文件 → 新 DoclingDocument
  2. 新 chunk_job + 新 chunks
  3. 旧 chunk_job → superseded
  4. 新 chunks → 自动触发向量化
  5. 旧 embedding_job → superseded

独立重新向量化 (POST /reembed)：
  1. 校验 parsed_document 归属
  2. 校验存在活跃 (非 superseded) chunk_job
  3. 创建新 embedding_job
  4. 旧 embedding_job → superseded
  5. 从已有 chunks 读取文本 → 生成向量 → 写入新 embeddings

模块 职责 建议位置
EmbeddingAdapter（接口） 定义 embed 接口契约：embed(texts: list[str]) → list[list[float]]。所有 Embedding 实现需遵循此接口。 backend/app/services/embedding_adapter.py
OllamaEmbeddingAdapter 封装 Ollama HTTP API 调用：向 {base_url}/api/embeddings 发送 POST 请求，解析返回向量。支持 batch 和 retry。 backend/app/services/embedding_adapter.py

DocumentEmbeddingService 编排向量化作业的创建、状态管理、chunk 文本读取、批量调用适配器、结果持久化。不直接调用 Embedding 模型。 backend/app/services/document_embedding.py
DocumentEmbeddingJob 记录向量化作业生命周期：状态、模型名、版本、维度、配置快照、chunk 数、错误信息。 backend/app/models/document_embedding.py
DocumentChunkEmbedding 存储单个 chunk 的向量（pgvector vector(N)），关联 chunk_id 和 embedding_job_id。 backend/app/models/document_embedding.py
分块服务扩展点 在 run_parse_job 中，分块成功后自动调用 DocumentEmbeddingService。沿用 make_document_chunking_service 的工厂模式。 backend/app/services/document_parse_dispatcher.py
向量化 API 路由 提供向量化作业查询、向量结果分页查询、重新向量化 API。 backend/app/api/routes/document_embedding.py
核心原则：EmbeddingAdapter 是项目中唯一接触 Embedding 模型的地方；服务层只接收项目内定义的向量结果；路由层只接收请求和返回 schema。适配器接口仅暴露 embed(texts) → vectors，屏蔽 Ollama/sentence-transformers/cloud API 的实现差异。数据库模型保存 knowra 的稳定契约，pgvector 类型只出现在模型层和 migration 中。

6. 数据模型设计

document_embedding_jobs — 向量化作业表

字段 类型 说明
id UUID PK 主键。
chunk_job_id UUID FK → document_chunk_jobs.id 关联分块作业。独立 /reembed 时，此字段指向当前活跃的分块作业。
parsed_document_id UUID FK → parsed_documents.id 冗余关联解析结果，便于从 parsed_document 维度直接查询向量化状态。
owner_user_id UUID FK → users.id 冗余当前用户归属，便于权限过滤。
status String(32) queued、running、succeeded、failed、superseded。superseded 表示被新参数的向量化作业取代，旧向量保留但不活跃。
embedder_name String(64) Embedding 模型名称，例如 bge-m3、qwen3-embedding。
embedder_version String(64) nullable Embedding 模型版本标识，便于结果可重复性分析。
embedding_dim Integer 向量维度，例如 1024。用于校验和 pgvector 类型对齐。
embed_config_json JSON nullable 向量化配置快照，例如 {"model": "bge-m3", "service": "ollama", "base_url": "http://localhost:11434", "batch_size": 32}。用于审计、对比和重新向量化。
chunk_count Integer 成功后写入的向量总数，应等于被向量化的 chunk 数。
attempt_count Integer 重试次数。
started_at / finished_at DateTime(tz) nullable 记录耗时。
error_code / error_message String nullable 可展示、可诊断的错误（如 Ollama 服务不可达、模型未找到、超时）。
created_at / updated_at DateTime(tz) 时间戳。

document_chunk_embeddings — 向量存储表

字段 类型 说明
id UUID PK 主键。
chunk_id UUID FK → document_chunks.id 关联被向量化的 chunk。一个 chunk 可以有多个 embedding（不同 embedding_job），查询时只返回活跃 job 的。
embedding_job_id UUID FK → document_embedding_jobs.id 关联向量化作业。
owner_user_id UUID FK → users.id 冗余当前用户归属。
embedding pgvector vector(N) 向量数据，N 由模型维度决定（首版 1024）。使用原始 SQL 创建此列（SQLModel 不原生支持 pgvector 类型）。
sequence_index Integer 冗余 chunk 序号，便于按文档顺序返回检索结果时使用。
created_at DateTime(tz) 时间戳。
为什么不直接在 document_chunks 表上加 vector 列？
  • 一个 chunk 可能需要多个不同模型的向量版本（通过 /reembed 切换模型）。
  • superseded 机制要求旧向量保留但不活跃，独立表更容易隔离活跃/非活跃数据。
  • chunk 和 embedding 的生命周期不同：chunk 随分块变更而变化，embedding 可以独立更新（不重新分块时）。
  • 独立表避免向量索引重建影响 chunk 查询性能。
pgvector SQLModel 集成方案：SQLModel 不原生支持 pgvector 类型。Model 中使用 Any = Field(sa_type=VECTOR(dim)) 声明 embedding 列（需要导入 pgvector 的 VECTOR）。Migration 中使用 op.execute("ALTER TABLE ... ADD COLUMN embedding vector(1024)") 原始 SQL。

7. Embedding 适配器设计

向量化适配器是项目中唯一接触 Embedding 模型/SDK 的模块。定义统一的 EmbeddingAdapter 接口，首版提供 OllamaEmbeddingAdapter 实现。

适配器接口契约

from abc import ABC, abstractmethod

class EmbeddingAdapter(ABC):
    """将文本列表转化为向量列表的统一接口。"""

    @abstractmethod
    def embed(self, texts: list[str]) -> list[list[float]]:
        """
        接收文本列表，返回相同长度的向量列表。

        text[i] 对应返回的 vectors[i]。
        每个 vector 的长度等于模型维度。

        Raises:
            EmbeddingServiceError: 服务不可达、模型未找到等。
            EmbeddingTimeoutError: 请求超时。
            EmbeddingBatchError: 批量请求失败（部分成功）。
        """
        ...

    @property
    @abstractmethod
    def dimension(self) -> int:
        """返回当前模型的 Embedding 维度。"""
        ...

    @property
    @abstractmethod
    def model_name(self) -> str:
        """返回当前模型的名称标识。"""
        ...

批量优化

Ollama 的 /api/embeddings 当前版本只支持单文本输入。如需批量优化，有两个方向：(1) 客户端并发请求（asyncio + aiohttp）；(2) 切换到支持批量输入的部署方式（如 sentence-transformers 直连或 vLLM embedding 端点）。首版按顺序单条请求即可——个人知识库文档量小，分块完成后后台任务中每条约 50-200ms（BGE-M3 on CPU），100 个 chunk 约 5-20 秒。如果觉得慢，后续可通过 concurrent.futures.ThreadPoolExecutor 并发调用 Ollama。

错误处理策略

错误场景 处理方式
Ollama 服务不可达 重试 3 次（指数退避），仍失败则标记 embedding_job 为 failed，error_code=service_unreachable。
模型未找到 不重试，标记 failed，error_code=model_not_found，提示用户执行 ollama pull bge-m3。
单条请求超时 重试 1 次，仍失败则标记 failed，error_code=request_timeout。已成功生成的向量不保存（原子性：全部成功或全部失败）。
向量维度不匹配 在适配器初始化时探测维度，与配置的 embedding_dim 比对；不匹配则抛错阻止写入。

8. API 设计

接口 用途 响应语义
GET /api/document-embedding-jobs/{job_id} 查询向量化作业状态（模型、维度、进度、错误）。 只允许当前用户读取自己的作业。404 如果不存在或无权限。
GET /api/parsed-documents/{parsed_document_id}/embedding-job 查询某解析结果的最新向量化作业状态。 返回最新（非 superseded）向量化作业；404 如果从未向量化。
GET /api/parsed-documents/{parsed_document_id}/embeddings?offset=0&limit=50 分页查询活跃向量化结果（含关联 chunk 的元数据）。 仅返回最新非 superseded 向量化作业的结果，按 sequence_index 排序。没有活跃作业时返回空页。
GET /api/document-chunks/{chunk_id}/embedding 查询单个 chunk 的活跃向量详情。 返回向量、维度、模型名称、创建时间。向量在 JSON 中以浮点数组返回。
POST /api/parsed-documents/{parsed_document_id}/reembed 使用新参数（模型、服务地址等）重新向量化，不重新分块。旧向量化作业标记为 superseded。 返回 202 Accepted 和新向量化作业；存在运行中作业返回 409；无可用的分块结果返回 412。

    batch_size: int | None = None          # 覆盖批处理大小

# 查询向量列表响应（分页）
class DocumentChunkEmbeddingRead(BaseModel):
    id: UUID
    chunk_id: UUID
    sequence_index: int
    embedding: list[float]  # 全量向量数组
    chunk_text: str | None  # chunk 的原始文本（便于调试）
    chunk_heading_path: list[str] | None
    chunk_page_numbers: list[int] | None
    created_at: datetime

class DocumentChunkEmbeddingPageRead(BaseModel):
    items: list[DocumentChunkEmbeddingRead]
    total: int
    offset: int
    limit: int
不提供独立的 POST /embed API：首次向量化始终由分块流程自动触发，与分块的自动触发策略一致。只有需要更换 Embedding 模型或参数时才通过 POST /reembed 手动触发重新向量化。重新分块（/rechunk）成功后自动触发新分块结果的向量化。

9. 作业执行策略

向量化在当前 run_parse_job 后台任务内扩展执行，分块成功后自动触发：

run_parse_job 执行流程（嵌入扩展）：

  # --- 现有流程（不变）---
  1. 解析原始文件 → DoclingDocument
  2. 持久化 parsed_documents + document_segments
  3. mark_parse_job_succeeded()

  # --- 分块（已有）---
  4. if document_chunking_enabled:
       a. 创建 chunk_job (queued → running)
       b. DoclingChunkerAdapter.chunk(docling_document)
       c. 持久化 document_chunks
       d. chunk_job → succeeded

  # --- 向量化（新增）---
  5. if document_embedding_enabled and chunk_job.succeeded:
       a. 创建 embedding_job (queued → running)
       b. 从 document_chunks 读取每个 chunk 的 contextualized_text
          （≤2KB 从 text 字段读，>2KB 从 storage_key 对应的文件读）
       c. EmbeddingAdapter.embed(texts) → vectors
       d. 逐条写入 document_chunk_embeddings
          （embedding 列使用 pgvector VECTOR 类型）
       e. embedding_job.chunk_count = len(vectors)
       f. embedding_job → succeeded
  6. 向量化失败不影响解析和分块结果（兜底 log + 作业状态标记）

chunk 文本读取策略

def get_contextualized_text(chunk: DocumentChunk, artifact_storage: ChunkArtifactStorage) -> str:
    """获取 chunk 的 contextualized_text，自动处理数据库/文件存储。"""
    if chunk.contextualized_text is not None:
        return chunk.contextualized_text  # ≤2KB，直接从 DB 读
    if chunk.contextualized_text_storage_key is not None:
        path = artifact_storage.path_for(chunk.contextualized_text_storage_key)
        return path.read_text(encoding="utf-8")  # >2KB，从文件读
    # fallback: 使用原始 text
    if chunk.text is not None:
        return chunk.text
    if chunk.text_storage_key is not None:
        path = artifact_storage.path_for(chunk.text_storage_key)
        return path.read_text(encoding="utf-8")
    raise MissingChunkTextError(f"Chunk {chunk.id} has no readable text")

独立重新向量化（POST /reembed）

reembed 执行流程：
  1. 校验 parsed_document 归属
  2. 获取当前活跃 chunk_job 及其 chunks
     → 如果没有活跃 chunk_job，返回 412 "无可用的分块结果"
  3. 检查是否有运行中的 embedding_job
     → 如果有，返回 409 "向量化作业已在运行中"
  4. 创建新 embedding_job (queued → running)
     → embedder_name、embedding_dim 等由请求或默认配置决定
  5. for each chunk in active_chunks:
       text = get_contextualized_text(chunk)
       vectors[i] = adapter.embed([text])[0]
       写入 document_chunk_embeddings
  6. embedding_job → succeeded
  7. 旧 embedding_job → superseded

10. 配置建议

配置项 默认值 说明

关键设计约束：
  • 向量化在分块成功的同一后台任务中自动执行，chunk text 从已持久化的 document_chunks 记录读取（而非内存直传——因为 chunk 文本可能 >2KB 走文件存储，内存中不一定有全文）。
  • document_chunks 表保持不变，不新增字段。向量存储完全独立于 chunk 表。
  • 重新分块（/rechunk）成功后自动触发新分块结果的向量化，旧向量化作业跟随旧分块作业被标记为 superseded。
  • 独立重新向量化（/reembed）只更换模型参数，不重新分块，旧向量化作业被标记为 superseded。
  • 分块失败或被禁用的文档不触发向量化。

12. 前端体验

状态 界面反馈 用户动作
分块成功，向量化自动进行 展示"分块完成，正在生成语义向量…"。 无需操作。向量化在分块后台任务内自动执行。
running 显示带进度提示的状态（"正在向量化 15/100…"）。 可刷新状态或稍后查看。
succeeded 展示"向量化完成" + 模型名称 + 维度 + chunk 数。 后续可进入语义检索和 RAG 问答流程。
向量化作业失败 展示可理解错误信息（"Ollama 服务未连接"、"模型未安装" 等）。 提供"重新向量化"入口；分块结果仍可用（不受影响）。
重新向量化（/reembed） 展示"重新向量化"入口，允许切换模型、调整配置。 触发使用新参数重新生成向量，旧向量保留但标记为 superseded。
分块中/分块失败 向量化状态不可用，需先完成分块。 无法触发向量化操作。
嵌入功能被禁用 展示"向量化未启用"提示。 分块仍然可用，但无向量化入口。
前端接口调用流：沿用 HomeView 中已建立的模式：
  1. 上传 → 解析 → 自动分块 → 自动向量化（一条链式轮询）
  2. 分块完成后，调用 GET /parsed-documents/{id}/embedding-job 轮询向量化状态
  3. 向量化完成后，调用 GET /parsed-documents/{id}/embeddings 获取向量预览
  4. 在分块面板中新增"向量化状态"区域，展示模型名称、维度、向量数、错误信息
  5. 重新分块按钮旁边新增"重新向量化"按钮（可选模型切换）
前端不承诺已完成语义检索或 RAG 问答——向量化完成只表示"文本已转化为可检索的语义向量"，实际检索和问答在后续能力中实现。

13. MVP 范围

纳入 MVP

  • 新增 OpenSpec 变更 add-document-embedding，新增能力规格 document-embedding。
  • 向量化作业表（document_embedding_jobs）和向量存储表（document_chunk_embeddings），含 pgvector vector(1024) 列。
  • Alembic migration（原始 SQL 创建 vector 列）。
  • EmbeddingAdapter 抽象接口 + OllamaEmbeddingAdapter 实现（连接本地 Ollama 服务，调用 bge-m3）。
  • DocumentEmbeddingConfig 配置 dataclass，支持 snapshot() 写入作业。
  • DocumentEmbeddingService 编排向量化作业创建、chunk 文本读取、批量调用适配器、结果持久化。
  • 分块成功后在同一后台任务中自动触发向量化（扩展 run_parse_job）。
  • 独立重新向量化 API（POST /parsed-documents/{id}/reembed），旧作业标记 superseded。
  • 向量化作业状态查询 API + 向量结果分页查询 API。
  • 权限校验：只能查看和操作自己的向量化结果。
  • 配置项：开关、模型名、维度、服务地址、超时、重试。

暂不纳入 MVP

  • 语义检索和 RAG 问答。
  • 混合检索（向量 + 全文 + 元数据过滤）。
  • 多模型向量并存（同时维护多套活跃向量）。
  • 向量化进度实时推送（WebSocket）。
  • 独立 worker + 队列的生产级部署。
  • sentence-transformers 直连适配器（首版仅 Ollama）。
  • 云 API 适配器（OpenAI/Cohere 等）。
  • 向量质量评估和检索质量评测。
  • 旧向量清理 API。
  • 批量并发调用 Ollama（首版顺序单条请求）。
  • reranking 模型集成。

15. 风险与开放问题

风险 影响 缓解方式
Ollama 服务依赖 Ollama 未启动或模型未下载时无法向量化。 ① 健康检查：后端启动时可探测 Ollama /api/tags 端点，log 警告而非崩溃。② 错误信息明确告诉用户执行 ollama pull bge-m3。③ 向量化作为可选增强，失败不影响分块结果可用性。④ DOCUMENT_EMBEDDING_ENABLED=false 可全局禁用。
向量化耗时不可控 CPU 模式下 BGE-M3 每条约 50-200ms，1000 条 chunk 需 50-200 秒。 ① 首版顺序单条请求，适合个人知识库（单文档通常 < 100 chunks，约 5-20 秒）。② 后续可并发请求（DOCUMENT_EMBEDDING_CONCURRENCY）或切换到 sentence-transformers 直连（一次 encode 批量文本，更快）。
向量维度与模型不匹配 切换模型后维度变更，pgvector 列类型必须匹配。 ① 首版固定 BGE-M3 (1024d)，Alembic migration 中写死 vector(1024)。② 如果后续需要不同维度，建议新增独立的 embedding 表或动态 DDL——但这是后续变更，不在首版范围内。③ /reembed 请求中如果指定了不同模型，校验其维度必须与当前表结构一致。
Ollama API 限流/不稳定 并发或高频请求可能被限流或超时。 ① 首版顺序单条请求（concurrency=1），不存在并发压力。② 重试 3 次（指数退避）。③ 超时配置可调。
长 chunk 文本读取性能 chunk 文本 >2KB 需从文件系统读取，可能增加 I/O。 ① 大多数 chunk 文本 ≤2KB（直接入库），只有极少数超长文本走文件。② 文件读取是顺序 I/O，影响可控。
BackgroundTasks 不具备生产级可靠性 与解析和分块作业相同的风险。 ① 与解析和分块一起，后续统一替换为独立 worker + 队列。② 向量化作业状态持久化，失败可诊断和重试。
Ollama embedding 端点仅支持单文本 无法批量输入，N 个 chunk 需 N 次 HTTP 请求。 ① 在个人知识库场景下可接受（几十到几百个 chunk）。② 后续可通过 concurrent.futures 并发请求优化。③ 或切换到 sentence-transformers 直连获得原生 batch 支持。

16. OpenSpec 建议

建议创建 OpenSpec 变更 add-document-embedding，新增能力规格 document-embedding。该变更应聚焦"在分块完成后将 chunk 文本转化为语义向量并持久化到 pgvector"，不要同时承诺语义检索、RAG 问答或混合检索。

建议 requirements

  • 系统 SHALL 在分块作业成功后为当前用户自动创建向量化作业，并持久化作业状态以便查询。
  • 系统 SHALL 在分块作业成功后，在同一后台任务中自动从已持久化的 chunk 记录读取 contextualized_text 并调用 Embedding 适配器生成向量。
  • 系统 SHALL NOT 提供独立的 POST /embed API；首次向量化始终由分块流程自动触发。
  • 系统 SHALL 支持用户通过 POST /parsed-documents/{id}/reembed API 使用自定义参数重新向量化（不重新分块）；旧向量化作业标记为 superseded。
  • 系统 SHALL 在重新分块（/rechunk）成功后自动触发新分块结果的向量化，旧向量化作业标记为 superseded。
  • 系统 SHALL 使用 EmbeddingAdapter 抽象接口封装 Embedding 模型调用，首版提供 OllamaEmbeddingAdapter 实现（调用 Ollama HTTP API）。
  • 系统 SHALL 将向量存储到 document_chunk_embeddings 表的 pgvector vector 列，维度与模型对齐（首版 BGE-M3 = 1024）。
  • 系统 SHALL 持久化向量化作业状态、配置快照、模型名称和维度、错误信息。
  • 系统 SHALL 在向量化作业标记为 superseded 时保留旧向量不主动删除，查询时仅返回最新（非 superseded）作业的向量。
  • 系统 SHALL NOT 修改已有的 document_chunks 表结构。
  • 系统 SHALL 防止用户读取不属于自己的向量化结果。
  • 系统 SHALL 处理 Ollama 服务不可达、模型未安装、请求超时等异常并给出可诊断的错误信息。
  • 系统 SHALL NOT 在首版实现语义检索、RAG 问答、混合检索或 reranking，也不实现 pgvector 高级索引（如 HNSW/IVFFlat）。

建议 tasks

  1. 创建 OpenSpec proposal、design、tasks 和 document-embedding 规格增量。
  2. Spike 验证：Ollama API 调用方式（/api/embeddings 端点）、BGE-M3 输出维度确认、pgvector 插入和查询测试。
  3. 新增 EmbeddingAdapter 抽象接口和 OllamaEmbeddingAdapter 实现（含维度探测、超时、重试）。
  4. 新增 DocumentEmbeddingConfig 配置 dataclass + Pydantic Settings 配置项。
  5. 新增 DocumentEmbeddingJob 和 DocumentChunkEmbedding SQLModel 模型（含 pgvector VECTOR 类型）。
  6. 新增 Alembic migration：创建 document_embedding_jobs 和 document_chunk_embeddings 表。
  7. 新增 Pydantic schemas（DocumentEmbeddingJobRead、DocumentChunkEmbeddingRead、ReembedRequest 等）。
  8. 实现 DocumentEmbeddingService（编排向量化作业创建、chunk 文本读取、批量调用适配器、结果持久化）。
  9. 在 run_parse_job 中扩展向量化逻辑：分块成功后自动触发。
  10. 实现向量化 API 路由：作业查询、向量分页查询、重新向量化；注册到 api_router。
  11. 实现前端向量化状态展示和重新向量化入口（新增 API 函数、类型、状态管理和 UI 组件）。
  12. 编写各层级测试（模型、配置、适配器、服务、API、集成）。
  13. 更新 .env.example、README 和端侧章程；运行 lint/test/build 验证。
