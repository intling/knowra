knowra 文件上传存储方案探索

本文沉淀“基于现有用户体系新增文件上传能力”的最终探索结论。范围限定为应用内原始文件存储与上传元数据管理，不包含文档解析、分块、向量索引、检索或 RAG 问答生成。

阶段：方案探索 推荐路径：先存储，后处理 用户归属：后端解析当前用户 不实现认证入口

1. 探索结论

推荐先实现“上传原始文件并在应用内保存”的最小能力：前端选择文件后通过 multipart/form-data 调用后端上传接口；后端使用现有 get_current_user(session) 解析当前用户，写入 uploaded_files 元数据，并把原始文件保存到后端可配置的本地存储目录。

这个边界与当前基础用户体系天然衔接：用户归属由后端决定，前端不提交 owner_user_id。后续接入登录、团队、文档解析或索引时，只需要复用同一份上传记录和文件存储键，不需要重写上传入口。

本阶段成功标准不是“上传后可问答”，而是“文件已经进入 knowra 的受控存储，并能明确追溯到当前用户、原始文件名、大小、类型、状态和存储位置”。

2. 为什么先只做存储

方案 内容 收益 代价 结论
推荐：上传并保存原始文件 保存文件二进制与元数据，状态为 stored。 范围清晰、容易测试、可直接支撑后续解析链路。 上传后暂不能基于文件问答。 当前最合适。
上传后同步解析 接口内完成存储、解析、分块和索引。 用户看起来更接近完整知识库闭环。 接口耗时长，错误分支复杂，会提前耦合解析器和索引设计。 暂不采用。
直接接对象存储 上传到 S3、OSS、MinIO 等外部存储。 更接近生产扩展形态。 当前会引入部署和凭据复杂度，影响本地开发闭环。 保留接口抽象，后续替换。

3. 推荐数据流

1. 选择文件 前端复用现有附件入口，生成 File 对象。
2. 上传请求 调用 POST /api/uploads，请求体为 multipart/form-data。
3. 用户解析 后端通过 get_current_user() 获取资源归属。
4. 文件落盘 使用系统生成的存储键写入本地目录。
5. 元数据入库 写入 uploaded_files 并返回上传记录。 
┌────────────┐
│ Frontend   │
│ File input │
└─────┬──────┘
      │ multipart/form-data
      ▼
┌──────────────────────────┐
│ POST /api/uploads         │
│ route: validate request   │
└─────┬────────────────────┘
      │ get_current_user(session)
      ▼
┌──────────────────────────┐
│ Upload service            │
│ - generate upload id      │
│ - generate storage key    │
│ - stream file to storage  │
│ - persist metadata        │
└─────┬──────────────┬─────┘
      │              │
      ▼              ▼
┌──────────────┐  ┌────────────────┐
│ Local storage │  │ uploaded_files │
│ original blob │  │ metadata       │
└──────────────┘  └────────────────┘

4. 后端边界草案

接口

接口 用途 说明
POST /api/uploads 上传单个文件 接收字段 file；可选接收 source 或 display_name，但不接收 owner_user_id。
GET /api/uploads 列出当前用户上传记录 只返回当前用户未删除记录，支持后续资料列表或调试。
GET /api/uploads/{id} 读取单条上传记录 校验记录属于当前用户；本阶段返回元数据，不直接开放文件下载也可以。
最小首批实现可以只包含 POST /api/uploads。如果前端需要展示“已上传资料”，再加入列表接口会更自然。

响应草案

{
  "id": "b3f50b2a-3b88-4a8b-a8d8-3e5d9fb1884a",
  "owner_user_id": "00000000-0000-0000-0000-000000000001",
  "original_filename": "course-notes.pdf",
  "content_type": "application/pdf",
  "byte_size": 482193,
  "storage_key": "uploads/00000000-0000-0000-0000-000000000001/b3f50b2a-3b88-4a8b-a8d8-3e5d9fb1884a/original.pdf",
  "status": "stored",
  "created_at": "2026-05-24T10:30:00Z",
  "updated_at": "2026-05-24T10:30:00Z"
}

响应中可以包含 owner_user_id 作为服务端事实，但请求中不能由前端提交它。后续真实认证接入后，这个原则仍然成立。

5. 数据模型草案

字段 类型倾向 说明
id UUID 上传记录主键。
owner_user_id UUID FK 关联 users.id，由后端当前用户解析结果写入。
original_filename String 用户上传时的文件名，仅用于展示，不用于真实存储路径。
content_type String nullable 客户端上报的 MIME 类型，可为空，后续可增加服务端嗅探。
byte_size BigInteger 文件大小，用于限制、展示和审计。
storage_key String unique 应用内稳定存储键，不暴露真实绝对路径。
checksum_sha256 String nullable 建议纳入，便于完整性校验和后续去重分析。
status String 初始支持 stored、failed、deleted；后续解析链路可扩展 processing、indexed。
error_message String nullable 上传存储失败时保存可诊断信息，避免只在日志中可见。
deleted_at DateTime nullable 保留软删除能力，与用户体系保持一致。
created_at / updated_at DateTime 审计与排序字段。

6. 存储策略

本阶段

使用后端本地目录作为应用内存储，例如配置项 upload_storage_dir 指向 backend/storage/uploads 或部署时挂载的数据卷。

文件路径由服务端生成，建议包含用户 ID 和上传 ID，避免直接使用原始文件名。

后续演进

抽象 FileStorage 服务接口，当前实现为 local storage；未来可替换为 MinIO、S3 或 OSS。

数据库只保存 storage_key，不保存绝对路径，从而降低存储后端切换成本。 

upload_storage_dir/
└── uploads/
    └── {owner_user_id}/
        └── {upload_id}/
            └── original{safe_extension}
原始文件名不能直接拼入真实路径。它可能包含路径分隔符、不可见字符或超长内容，应只作为展示字段保存。

7. 前端边界草案

前端目前已经有文件选择入口和选中文件提示。推荐在此基础上新增上传提交状态，而不是先做复杂资料管理页。

区域 行为 用户反馈
附件按钮 选择单个文件。 显示文件名、大小，可移除。
发送/上传动作 如果有文件，先调用上传接口。 显示上传中、成功、失败。
用户不可用 /api/users/me 失败时禁用上传。 提示当前用户不可用，避免创建无归属文件。
API client 新增 multipart 上传封装。 保留 HTTP 状态和错误 detail，页面层转成中文反馈。

本阶段不要求把文件和问题一起组成 RAG 请求。用户选择文件后的动作可以先表达为“上传资料”，成功后再让后续变更接入解析与提问链路。

8. 错误与限制

场景 推荐响应 说明
当前用户不可用 503 与现有 /api/users/me 语义一致，不创建无归属资源。
没有文件字段 422 由 FastAPI 请求校验或路由层明确返回。
文件为空 400 避免保存无意义记录。
超过大小限制 413 大小限制通过配置控制，例如 max_upload_bytes。
不支持的类型 415 首批可以允许常见文档类型，或暂只限制大小。
落盘失败 500 不应留下指向不存在文件的成功记录；必要时记录 failed 状态供诊断。
大文件上传的流式处理是主要工程风险。实现时应避免一次性读取全部内容进入内存，并围绕大小限制、部分写入失败、事务一致性写测试。

9. OpenSpec 建议

这个探索已经足够进入一个独立 OpenSpec 变更，建议变更名使用 add-file-upload-storage。变更应聚焦在“上传原始文件并保存元数据”，不要同时承诺解析和索引。

建议能力规格

  • file-upload-storage：定义上传接口、文件元数据、应用内存储、当前用户归属、错误处理和前端反馈。

建议验收场景

  • 当前用户可用时，上传有效文件返回上传记录并写入元数据。
  • 上传记录的 owner_user_id 来自后端当前用户解析，而不是客户端请求字段。
  • 原始文件保存到应用内存储目录，数据库保存稳定 storage_key。
  • 当前用户不可用时，上传失败且不保存文件。
  • 超出大小限制或空文件上传时，返回明确错误。
  • 前端上传过程中展示加载、成功和失败反馈。

建议非目标

  • 不实现登录、注册、团队权限或用户切换。
  • 不实现文档解析、分块、embedding、向量索引或 RAG。
  • 不实现对象存储接入，但保留通过存储服务接口替换的设计空间。

10. 推荐下一步

  1. 创建 OpenSpec 变更 add-file-upload-storage，把本文结论转成 proposal.md、design.md、tasks.md 和规格增量。
  2. 实现时按 TDD 推进：先写后端上传 API、模型、服务和前端上传状态的失败测试，再写最小实现。
  3. 上传存储完成并归档后，再单独开启“文档解析与索引”变更，让解析链路消费 uploaded_files。
最终推荐：先把文件可靠地纳入 knowra 的用户归属和应用存储边界，再让后续知识处理链路基于这份稳定记录展开。
