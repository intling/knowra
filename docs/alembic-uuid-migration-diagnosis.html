Alembic UUID 迁移失败原因诊断

失败发生在 20260523_0001_create_users.py 创建 users 表后的默认用户插入步骤，不是 PostgreSQL 扩展安装问题，也不是模型层把字段定义成了字符串。

数据库列类型

users.id 是 UUID

迁移第 25 行使用 sa.Uuid() 创建主键列，因此 PostgreSQL 侧列类型是 uuid。

模型层也使用 UUID，方向一致。
插入参数类型

绑定参数被当成 VARCHAR

默认用户 ID 常量在迁移里是字符串，并通过原始 sa.text() 绑定。SQLAlchemy 无法从目标列推断该参数应为 UUID，于是 psycopg 执行时生成了 %(id)s::VARCHAR。

PostgreSQL 拒绝把该表达式写入 uuid 列。
报错链路

从迁移代码到数据库错误

  1. 迁移创建 users.id uuid。
  2. 迁移执行原始 SQL： INSERT INTO users (id, ... ) VALUES (:id, ...)。
  3. DEFAULT_USER_ID 在迁移文件中是字符串： "00000000-0000-0000-0000-000000000001"。
  4. 绑定后的语句中参数类型变为 VARCHAR，日志显示 VALUES (%(id)s::VARCHAR, ...)。
  5. PostgreSQL 报出： column "id" is of type uuid but expression is of type character varying。
关键代码

触发点

DEFAULT_USER_ID = "00000000-0000-0000-0000-000000000001"

op.execute(
    sa.text(
        """
        INSERT INTO users (id, display_name, status, created_at, updated_at)
        VALUES (:id, 'Default User', 'active', CURRENT_TIMESTAMP, CURRENT_TIMESTAMP)
        """
    ).bindparams(id=DEFAULT_USER_ID)
)
根因

原始 SQL 绑定参数缺少 UUID 类型信息

这是迁移脚本的参数类型绑定问题。表定义、运行时模型和服务层都以 UUID 为准，但迁移里的种子数据插入绕过了 ORM/SQLModel 的类型映射， 直接把字符串参数传给 uuid 列，导致 PostgreSQL 类型检查失败。
