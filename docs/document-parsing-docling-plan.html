knowra 文档解析功能方案（Docling）

本文沉淀“在现有上传能力之后，新增基于 Docling 的文档解析能力”的方案。范围覆盖 PDF、DOCX、PPTX、TXT、Markdown 的接入、解析作业、解析结果存储、状态反馈和后续 RAG 扩展边界；不在本文中实现代码。

阶段：方案探索 日期：2026-06-04 推荐变更：add-document-parsing-docling 解析核心：Docling DocumentConverter 产物策略：不保存派生图片 执行策略：先 BackgroundTasks，后 worker 队列 前置能力：file-upload-storage 已归档

1. 方案结论

推荐把“文档解析”设计成上传记录之后的异步处理链路：上传 API 只负责保存原始文件；解析 API 或解析调度器读取 uploaded_files.storage_key，创建解析作业，调用 Docling 转换为统一的 DoclingDocument，再保存 Markdown、纯文本、Docling JSON 和结构化元数据。首版不保存 PDF 页面图、图表图片、表格截图等派生图片资产；执行上先使用 FastAPI BackgroundTasks 验证闭环，同时让服务边界兼容后续独立 worker + 队列。

这样做能让 knowra 先获得一个稳定的资料处理边界：用户上传的原始文件可追溯，解析结果可重建，后续分块、embedding、向量索引、引用定位和问答生成都可以消费同一份解析结果。

首个版本的成功标准不是“上传后立刻可问答”，而是“文档被可靠解析为可预览、可追溯、可供后续索引的标准中间结果”。

2. 现有基础与约束

项目现状 对解析方案的影响
已有 uploaded_files 表，包含 owner_user_id、storage_key、checksum_sha256、status 和错误字段。 解析结果应引用上传记录，不重复保存原始文件路径，不由前端提交用户归属。
上传服务当前把文件写入本地可配置目录，storage_key 由服务端生成。 Docling 适配器从存储服务解析出本地路径或临时文件，而不是相信客户端文件名。
当前上传白名单由 ALLOWED_UPLOAD_CONTENT_TYPES 配置管理。 新增解析能力不应默认放宽上传白名单；PPTX 等格式是否可上传由部署配置显式开启，解析入口再通过独立格式策略二次校验。
项目章程要求新功能经 OpenSpec 管理，并按 TDD 推进。 下一步应先创建 OpenSpec 变更，再从失败测试驱动作业状态、权限、解析适配和 API 行为。

3. Docling 能力匹配

Docling 官方文档说明它可以把多种输入格式转换为统一的 DoclingDocument，并导出 Markdown、JSON、Text、HTML 等格式。DocumentConverter 是主要入口，支持限制允许格式、设置 PDF pipeline 选项、设置最大文件大小和最大页数。

目标类型 建议输入识别 Docling 处理策略 MVP 备注
PDF application/pdf，扩展名 .pdf 使用 InputFormat.PDF，PDF OCR 和表格结构提取通过配置控制；不导出或保存页面图、图表图片、表格截图。 默认优先文本型 PDF；扫描件 OCR 可配置开启，避免首版解析耗时不可控。
DOCX application/vnd.openxmlformats-officedocument.wordprocessingml.document 使用 InputFormat.DOCX，导出 Markdown 和 Docling JSON。 适合作为首批稳定样例格式。
PPTX application/vnd.openxmlformats-officedocument.presentationml.presentation 使用 InputFormat.PPTX，保留页/幻灯片序号用于引用。 上传是否允许 PPTX 由配置显式开启；解析前需验证 OOXML 容器结构。
Markdown text/markdown、text/x-markdown，扩展名 .md 使用 InputFormat.MD 或字符串转换入口。 可作为解析链路最轻量的端到端测试样例。
TXT text/plain，扩展名 .txt 优先通过当前 Docling 版本验证 plain text 支持；若安装版本未暴露 TXT 输入枚举，则在同一解析适配层内用轻量文本读取兜底，并输出同样的规范化结果。 TXT 不应阻塞整体方案；关键是输出契约一致。
实施前建议做一个小型 spike：安装项目锁定版本的 Docling，分别用 5 个最小样例文件验证 DocumentConverter 的输入格式识别、输出字段、异常类型、耗时和 Python 3.14 兼容性。

4. 推荐数据流

1. 上传完成 原始文件已保存，uploaded_files.status 为 stored。
2. 创建作业 用户或系统触发解析，写入 document_parse_jobs。
3. 读取原文件 通过 storage_key 定位原始文件，校验用户归属和 checksum。
4. Docling 转换 调用 DocumentConverter 得到 DoclingDocument。
5. 保存结果 写入 Markdown、Text、Docling JSON、元数据和结构片段。
6. 对外反馈 前端展示解析状态、预览、错误信息和后续可索引状态。 
uploaded_files
  id, owner_user_id, storage_key, checksum_sha256
        |
        v
document_parse_jobs
  queued -> running -> succeeded | failed
        |
        v
DoclingDocument
        |
        +--> parsed_documents
        |     markdown_storage_key
        |     text_storage_key
        |     docling_json_storage_key
        |
        +--> document_segments
              sequence_index, page_no, heading_path, text, metadata

5. 后端模块边界

模块 职责 建议位置
DocumentParsingService 编排上传记录读取、作业状态流转、解析器调用、结果持久化和错误处理。 backend/app/services/document_parsing.py
DoclingParserAdapter 封装 Docling SDK，只暴露项目内的 ParsedDocument 结果，不把第三方类型扩散到路由层和模型层。 backend/app/services/parsers/docling.py
ParsedArtifactStorage 保存解析产物，例如 Markdown、Text 和 Docling JSON，沿用 storage_key 抽象；不保存 PDF 页面图、图表图片或表格截图。 复用或扩展上传存储服务
DocumentParseJob 记录解析任务生命周期、重试次数、错误、耗时和解析器版本。 backend/app/models/document_parse_job.py
ParsedDocument 记录一次成功解析的规范化产物和文档级元数据。 backend/app/models/parsed_document.py
DocumentSegment 保存解析后的可引用结构单元，后续可被 chunker、embedding 和引用展示复用。 backend/app/models/document_segment.py
核心原则：路由层只接收请求和返回 schema；Docling SDK 调用只能出现在适配器内；数据库模型保存 knowra 的稳定契约，而不是直接暴露第三方 SDK 内部结构。

6. 数据模型草案

document_parse_jobs

字段 说明
id UUID 主键。
uploaded_file_id 引用 uploaded_files.id。
owner_user_id 冗余当前用户归属，便于权限过滤和查询。
status queued、running、succeeded、failed、cancelled。
parser_name / parser_version 记录 docling 和实际包版本，便于结果可重复性分析。
attempt_count 支持失败重试和诊断。
started_at / finished_at 记录耗时和状态观测。
error_code / error_message 保存可展示、可诊断的错误，不把完整堆栈返回给前端。

parsed_documents

字段 说明
id UUID 主键。
uploaded_file_id / parse_job_id 关联原文件和成功作业。
source_checksum_sha256 记录本次解析对应的原文件 checksum，原文件变化时可判断结果是否过期。
markdown_storage_key 保存给预览、轻量分块和调试使用的 Markdown。
text_storage_key 保存纯文本，便于全文搜索、快速预览和低成本回退。
docling_json_storage_key 保存无损 Docling JSON，便于未来重新分块或提取表格、图片、层级。
title、page_count、metadata_json 保存文档级元数据，字段按解析结果逐步扩展。

document_segments

首版可以只保存粗粒度结构单元，例如标题、段落、表格、幻灯片文本、页内文本块。后续真正的 embedding chunks 可以在单独索引变更中从 document_segments 或 Docling JSON 派生。

7. 解析产物存储

推荐把解析产物和原始文件一样纳入应用内存储，但使用不同前缀，避免把大 JSON、Markdown 或纯文本直接塞入关系型数据库。解析产物只保存文本与结构化中间表示，不在本地保存 PDF 页面图、图表图片、表格截图或其他派生图片资源。

明确决策：解析链路保留原始文件、Docling JSON、Markdown 和纯文本；不保存页面渲染图、图表图片、表格图片等本地派生图片。后续如果需要引用图片，应从原始文件或外部对象存储策略重新设计，而不是在本阶段落本地图片缓存。 
storage/
└── parsed/
    └── {owner_user_id}/
        └── {uploaded_file_id}/
            └── {parse_job_id}/
                ├── content.md
                ├── content.txt
                └── docling.json
产物 用途 是否 MVP 必需
content.md 前端预览、轻量调试、人工检查解析质量。 是
content.txt 快速全文预览、后续简单分块或全文搜索。 是
docling.json 保留结构化中间表示，后续可重新生成 chunks、引用坐标和表格结构。 是
这个取舍会降低本地存储膨胀和隐私暴露面，但也意味着首版引用定位主要落在文件、页码、标题路径、段落序号和 Docling 结构引用上，不承诺可视化框选图片区域。

8. API 草案

接口 用途 响应语义
POST /api/uploads/{upload_id}/parse 为当前用户的已上传文件创建解析作业。 返回 202 Accepted 和解析作业；如果已存在运行中作业，返回 409 或已有作业。
GET /api/document-parse-jobs/{job_id} 查询解析状态。 只允许当前用户读取自己的作业，返回状态、错误、时间戳。
GET /api/uploads/{upload_id}/parsed-document 读取当前上传文件的最新成功解析结果。 返回解析元数据、可预览 Markdown 摘要、segment 数量和产物引用。
GET /api/parsed-documents/{id}/segments 分页读取结构片段。 供前端预览、调试和后续引用展示使用。
最小版本可以只做 POST /api/uploads/{upload_id}/parse 和 GET /api/document-parse-jobs/{job_id}，再让上传列表透出 parse_status。解析结果预览接口可以作为第二步。

9. 作业执行策略

策略 优点 代价 结论
同步解析 实现最少，接口返回即可知道结果。 PDF、OCR、PPTX 可能耗时过长，容易阻塞请求和超时。 不推荐作为默认。
FastAPI BackgroundTasks 适合当前 MVP，部署简单，可直接跟现有 FastAPI 服务集成。 进程重启会中断正在执行的任务，多实例部署时不能天然保证任务分发和重试。 本阶段采用，但必须用数据库作业表作为事实源，任务函数保持幂等，避免把业务状态只放在内存里。
独立 worker + 队列 可靠、可限流、可重试，适合 CPU 密集解析。 需要 Redis/RQ/Celery/Arq 等基础设施和运维约束。 作为上线后的演进方向；首版接口、模型和服务边界需提前兼容 worker 替换。
推荐路径：首版执行策略明确选择 FastAPI BackgroundTasks，以便快速完成上传后解析闭环；同时把 document_parse_jobs 作为唯一状态事实源，并把实际解析逻辑封装成可被 BackgroundTasks 或未来 worker 调用的 run_parse_job(job_id)。上线后如果出现多实例、长任务、失败重试或资源隔离需求，再把调度层替换为独立 worker + 队列，API 和数据模型保持不变。

BackgroundTasks 首版约束

  • API 只负责创建 queued 作业并提交后台任务，立即返回 202。
  • 后台任务启动后从数据库重新读取作业和上传记录，不依赖请求上下文里的 ORM 对象。
  • 任务执行前用状态更新把 queued 切到 running，执行完成后写入 succeeded 或 failed。
  • 任务函数必须支持重复调用：如果作业已成功，直接跳过；如果原文件不存在或无权限，写失败状态。
  • 部署上线前增加“启动时扫描 stuck jobs”的修复策略，例如把长时间 running 的作业标记为 failed 或重新入队。

上线演进边界

POST /api/uploads/{upload_id}/parse
  -> create document_parse_jobs row
  -> enqueue job id

MVP enqueue:
  BackgroundTasks.add_task(run_parse_job, job_id)

Production enqueue:
  queue.push(job_id)
  worker.run(run_parse_job, job_id)

10. 配置建议

配置项 示例 说明
PARSED_ARTIFACT_STORAGE_DIR storage/parsed 解析产物存储根目录。
DOCUMENT_PARSING_ENABLED true 部署开关，便于在依赖未就绪环境中禁用解析。
DOCUMENT_PARSE_MAX_BYTES 52428800 解析层最大文件大小，可与上传限制相同或更严格。
DOCUMENT_PARSE_MAX_PAGES 200 限制 PDF/PPTX 等页数，防止单任务耗尽 CPU。
DOCLING_PDF_OCR_ENABLED false 首版建议默认关闭 OCR，后续对扫描件按需开启。
DOCLING_ARTIFACTS_DIR storage/docling-artifacts 用于模型权重或 Docling 运行缓存，具体接法以锁定 Docling 版本为准；不用于保存页面图、图表图片或表格截图。
DOCUMENT_PARSE_DISPATCHER background_tasks 首版使用 FastAPI BackgroundTasks；后续可切换为 queue 以接入独立 worker。
DOCUMENT_PARSE_ALLOWED_CONTENT_TYPES PDF、DOCX、PPTX、TXT、Markdown 解析入口独立允许列表，不等同于上传默认白名单。
DOCUMENT_PARSE_ALLOWED_EXTENSIONS .pdf,.docx,.pptx,.txt,.md 配合 MIME、文件头、OOXML 容器和文本采样做轻量服务端识别。
ALLOWED_UPLOAD_CONTENT_TYPES 按部署显式配置 不因解析能力默认放宽；需要 PPTX 上传时由环境配置显式加入。

11. 前端体验

前端不需要在首版做复杂资料库页面。可以在已有上传入口之后展示解析状态，让用户知道资料已经进入处理流程。

状态 界面反馈 用户动作
stored 上传完成，等待解析。 可点击“解析”或由系统自动触发。
queued / running 显示解析中，避免重复触发同一文件解析。 可刷新状态或稍后查看。
succeeded 展示解析完成，可预览文本。 后续可进入分块、索引或提问流程。
failed 展示可理解错误，例如文件格式不支持、文件损坏、页数超限。 允许重试或重新上传。

12. MVP 范围

纳入 MVP

  • 新增 OpenSpec 变更 add-document-parsing-docling。
  • 支持 PDF、DOCX、PPTX、Markdown、TXT 的解析入口，并通过独立格式策略校验；上传白名单不默认随解析目标放宽。
  • 解析作业表和状态查询。
  • Docling 适配器，输出 Markdown、Text、Docling JSON。
  • 解析结果元数据和粗粒度 segments。
  • FastAPI BackgroundTasks 测试版 dispatcher，用于验证异步解析闭环。
  • 权限校验：只能解析和查看当前用户文件。
  • 基础前端状态展示：等待解析、解析中、完成、失败。

暂不纳入 MVP

  • 向量 embedding 和 pgvector 索引。
  • RAG 问答生成。
  • 复杂 OCR 多语言配置界面。
  • 保存 PDF 页面图、图表图片、表格截图或其他派生图片资源。
  • 图片、公式、表格的专门编辑或可视化。
  • 独立 worker + 队列的生产级部署。
  • 文档版本对比和增量解析。

13. 测试与验收建议

实现时应严格按项目 TDD 约束推进：先写失败测试，再写最小实现，再重构。解析类测试要尽量使用小型 fixture，避免把大文件和慢 OCR 纳入默认测试集。

测试层级 建议场景
模型测试 作业表字段、外键、状态索引、解析结果与上传记录关系。
服务测试 当前用户只能解析自己的文件；缺失原文件失败；重复解析返回已有作业或拒绝。
适配器测试 用 mock Docling 验证调用参数、产物写入、异常转换，并断言不会写入页面图、图表图片、表格截图等派生图片；另用极小 MD/TXT fixture 做真实转换 smoke test。
调度器测试 验证 BackgroundTasks dispatcher 会注册 run_parse_job(job_id)，API 立即返回 202，作业状态由持久化表驱动，后续可替换为队列 dispatcher。
API 测试 POST /api/uploads/{id}/parse 返回 202；无权限返回非 2xx；查询作业返回状态。
前端测试 上传后触发解析、解析中禁用重复提交、失败展示错误、成功展示可预览状态。
默认验证命令仍应包括后端 uv run ruff check .、uv run ruff format --check .、uv run pytest；前端按 front/AGENTS.md 的 lint/test/build 约束执行。

14. 风险与开放问题

风险 影响 缓解方式
PDF OCR 和模型依赖较重 安装慢、首次运行慢、CPU 占用高。 首版默认关闭 OCR；提供配置开关；把扫描件解析质量作为后续增强。
Docling 输出结构随版本演进 直接持久化第三方对象会带来迁移成本。 保存原始 Docling JSON，同时把 knowra 需要的字段规范化到自有表。
大文件和损坏文件 任务卡住、内存飙升或解析失败。 设置最大字节数、最大页数、超时、失败状态和重试次数。
TXT 支持细节需按版本确认 如果当前 SDK 没有显式 TXT 输入格式，强行走 Docling 会拖慢实现。 在 parser facade 下做版本探测和轻量兜底，保持输出契约一致。
引用定位还未完整设计 后续 RAG 引用可能只能指向段落或页，而不是精确坐标。 segments 先保存页码、序号、标题路径和 Docling 引用元数据；不依赖页面截图作为引用证据。
BackgroundTasks 不具备生产级可靠性 进程重启、滚动部署或多实例场景下可能中断正在执行的解析任务。 首版仅作为测试版执行策略；作业状态必须持久化，调度入口抽象为 dispatcher，后续替换为独立 worker + 队列。

15. OpenSpec 建议

建议下一步创建 OpenSpec 变更 add-document-parsing-docling，新增能力规格 document-parsing。该变更应聚焦“从已上传文件生成可追溯解析结果”，不要同时承诺 embedding、检索或 RAG。

建议 requirements

  • 系统 SHALL 为当前用户已上传文件创建文档解析作业。
  • 系统 SHALL 使用 Docling 作为主要解析器处理 PDF、DOCX、PPTX、Markdown，并处理 TXT 的规范化输出。
  • 系统 SHALL 使用独立解析允许列表和轻量服务端识别校验解析候选文件，不依赖上传默认白名单覆盖解析目标格式。
  • 系统 SHALL 持久化解析作业状态、错误、解析器版本和产物位置。
  • 系统 SHALL 保存 Markdown、纯文本、Docling JSON 和文档级元数据。
  • 系统 SHALL NOT 在首版保存 PDF 页面图、图表图片、表格截图等派生图片资产。
  • 系统 SHALL 在首版使用 FastAPI BackgroundTasks 调度解析作业，并保留替换为独立 worker + 队列的服务边界。
  • 系统 SHALL 防止用户解析或读取不属于自己的上传文件。
  • 前端 SHALL 展示解析状态，并在失败时保留重试可能性。
