🎯 下一步优先级：文档解析服务

knowra 的核心价值链是 接入资料 → 解析内容 → 分块索引 → 语义检索 → 带引用回答。

当前只有"接入资料"（文件上传）完成。文档解析是打通后续所有环节的关键节点：没有文本提取，就无法分块、无法嵌入、无法检索、无法回答。

建议按以下依赖链逐步推进，每个阶段遵循 OpenSpec + TDD 流程：

✓ 文件上传 → 文档解析 → 分块与索引 → 向量嵌入 → 语义检索 → RAG 问答 & 引用展示

🗄️ 中间态数据存储方案：解析后、分块前

在 RAG 管道中，解析后的原文是体量大、需持久、需溯源的中间产物。存错了地方，后续分块和重处理都会踩坑。

核心矛盾

knowra 当前已有清晰的存储模式：文件存磁盘 + 元数据存数据库：

LocalFileStorage (磁盘) uploads/{uid}/{fid}/original.pdf ← → PostgreSQL uploaded_files 表 (元数据)

解析后会产生两种性质不同的数据：

数据类型典型体量访问模式举例
原始提取文本 大（50KB - 5MB） 顺序读、少量写、极少更新 PDF 全文、Markdown 原文
解析元数据 小（几十 - 几百字节） 随机查询、过滤、状态追踪 标题、语言、页数、解析器版本

把大文本塞进 PostgreSQL 会拖慢查询和备份；只存磁盘不留元数据则丢了溯源和状态管理。

推荐方案：续用已有模式

解析的中间态数据应沿袭 LocalFileStorage + PostgreSQL 元数据的同一模式：

磁盘文件结构：
storage/uploads/{owner_user_id}/{upload_id}/
├── original.pdf  ← 已有：上传的原始文件
└── parsed/
├── plain_text.txt  ← 新增：提取的纯文本
└── metadata.json  ← 新增：结构化元信息（标题层级等）

数据库：新增 parsed_documents 表

content_key                str — 磁盘路径指针

为什么原文放磁盘、元数据放数据库？

📄 纯文本 → 磁盘

  • 200 页 PDF 可能提取 2-5MB 文本，PostgreSQL TOAST 膨胀 5-10×
  • 纯文本不需要 WHERE 或 JOIN，只被分块服务顺序读入
  • 重分块时直接读磁盘文件，无需重解析原始 PDF

🏷️ 元数据 → PostgreSQL

  • status 驱动 UI 展示和 API 过滤
  • 支持快速查询：WHERE status='parsed'、WHERE language='zh'
  • 外键 + 解析器版本形成完整溯源链
  • 校验和判断是否需要重解析

完整数据流

上传文件
↓

磁盘: uploads/{uid}/{fid}/original.pdf
↓ 触发解析
解析服务  parser_name 按 MIME 类型选择
  ├─ application/pdf   → pypdf
  ├─ text/markdown     → markdown
  ├─ application/vnd.openxmlformats-officedocument.wordprocessingml.document   → python-docx

  └─ text/plain        → 直接读取
↓ 产出

分块服务 读取 plain_text.txt → 切割为 chunks
↓ 产出

↓ 触发嵌入
pgvector  document_chunks.embedding

为什么 chunk 文本可以放数据库？

一个 chunk 通常 500-1000 字符（~1-2KB），即使一个文档产生 200 个 chunk，总共也就 200-400KB。这个量级在 PostgreSQL 完全没有问题 — 它们需要被 document_id 过滤、按 position 排序、按 embedding 做向量搜索，数据库是最佳归宿。唯有原始提取全文（可能数 MB）只需被分块服务顺序读取一次。

重处理：校验和驱动

✅ 改分块参数（chunk_size）

校验和一致 → 跳过解析，直接从磁盘读 plain_text.txt 重新分块

🔄 源文件更新

校验和不匹配 → 重新解析，生成新的 plain_text.txt 和元数据

与主流框架对比

框架原文存储元数据存储knowra 对应

knowra 的方案与主流框架推荐模式一致：大文本走磁盘，元数据走关系数据库。

何时升级为对象存储？

当前本地磁盘 + PostgreSQL 方案在以下场景足够：个人或小团队使用（数十 GB）、单机部署、不需要跨实例共享。当出现多实例部署、存储量超单机容量、或需要跨区域访问时，LocalFileStorage 可升级为 S3FileStorage（保持相同接口），无需改动业务逻辑。

1 文档解析服务 高优先级

为什么是第一步

文件上传只存储了原始二进制文件（LocalFileStorage），没有提取文本内容。解析是从"文件存储"到"知识可用"的分水岭。所有后续的 chunking、embedding、检索和 RAG 都依赖于此。

功能目标

  • 从 uploaded_files 读取已上传的原始文件
  • 根据 MIME 类型选择解析器，支持初始格式：纯文本 (.txt)、Markdown (.md)、PDF (.pdf)、DOCX (.docx)、PPTX (.pptx)
  • 提取纯文本内容，保留基本结构信息（标题层级、段落分隔）
  • 将解析结果持久化到数据库，关联原始文件记录
  • 解析失败时记录错误信息，标记文件状态为 parse_failed

建议的数据模型

  • 新增 parsed_documents 表：关联 uploaded_file_id，存储提取的纯文本、解析状态、字符数、行数等元数据
  • 可选择在同一个表中存储简单元信息（标题、语言检测等）

技术选型建议

  • PDF 解析：pypdf — 纯 Python 实现、轻量无依赖、社区活跃、支持文本提取
  • Markdown 解析：markdown（pip install markdown）— 成熟的 Python Markdown 库
  • DOCX 解析：python-docx — 提取 DOCX 文件段落文本和表格内容
  • PPTX 解析：python-pptx — 提取 PPTX 幻灯片文本和备注
  • 纯文本：直接读取，UTF-8 解码
  • 设计为可插拔解析器适配器模式，便于后续扩展 EPUB 等更多格式

API 设计建议

  • POST /api/uploads/{upload_id}/parse — 触发解析
  • GET /api/uploads/{upload_id}/parsed-content — 获取解析结果
  • 或选择上传后自动触发解析（取决于产品决策）

验收标准

  • TDD: 先写 RED 测试覆盖各格式解析成功与失败场景
  • 上传 txt/md/pdf 后可提取纯文本并持久化
  • 解析失败时有明确错误信息和状态回退
  • Alembic 迁移可正向和反向执行
  • 后端质量门禁全过（ruff + pytest）

2 文本分块与元数据索引 高优先级

为什么紧随解析之后

完整的文档文本太长，无法直接用于语义检索和 LLM 上下文窗口。分块是连接"文档"到"可检索单元"的桥梁。

功能目标

  • 将解析后的文档文本按语义边界切分为固定大小的 chunk（建议初始窗口 500-1000 字符，重叠 100-200 字符）
  • 每个 chunk 保留与原始文档的关联（文档 ID、在原文中的位置偏移）
  • 新增 document_chunks 表或类似模型存储分块结果
  • 存储 chunk 元数据：所属文档、位置索引、字符范围等

技术选型建议

  • 初始方案：简单的滑动窗口分块（按字符数 + 重叠）
  • Markdown 文件可按标题层级做语义分块（先按 # 标题分割，超长段落再用滑动窗口）
  • 设计为可替换的分块策略接口，为后续智能分块（语义边界检测、句子级分块）留出空间

验收标准

  • 解析完成后的文档可自动触发分块
  • chunk 长度和重叠参数可配置
  • 分块后可通过 API 查询文档的 chunk 列表
  • Alembic 迁移 + 全质量门禁通过

3 向量嵌入服务 中优先级

功能目标

  • 为每个 chunk 生成向量嵌入（embedding）
  • 将嵌入存储到 PostgreSQL 的 pgvector 列中（已通过迁移启用的 vector 扩展）
  • 增量为新建的 chunk 生成嵌入，避免全量重算
  • 嵌入模型可配置（通过环境变量指定模型 API endpoint 和模型名称）

技术选型建议

  • 嵌入模型：初期可使用 OpenAI text-embedding-3-small 或 text-embedding-3-large，通过配置可切换为本地部署模型（如 BGE、M3E）
  • 存储：在 document_chunks 表新增 embedding vector(1536) 列（维度取决于模型）
  • 为嵌入列创建 HNSW 索引以加速近似最近邻搜索
  • 设计模型适配器接口：统一 embed(texts) → vectors 协议，隔离具体模型调用

验收标准

  • 分块完成后可触发嵌入生成
  • 嵌入结果持久化到数据库并可通过 HNSW 索引高效检索
  • 模型切换仅需配置变更，无需改代码
  • 嵌入失败（模型 API 不可用）有明确错误处理

4 语义检索服务 中优先级

功能目标

  • 接受用户问题，生成问题嵌入
  • 在向量空间中检索最相似的 top-K 个 chunk
  • 返回匹配的 chunk 文本、来源文件、位置偏移等信息
  • 为后续 RAG 流程提供结构化的检索上下文

API 设计建议

  • POST /api/search — 接受查询文本和可选参数（top_k, similarity_threshold 等）
  • 返回结果包含 chunk 文本、来源文件名、置信度分数和位置信息

验收标准

  • 给定一个问题文本，返回语义相关度最高的 chunk 列表
  • 检索结果包含来源溯源信息（文件名、段落位置等）
  • top_k 和相似度阈值可通过参数调整

5 RAG 问答与引用展示 中优先级

功能目标

  • 将检索到的 chunk 上下文 + 用户问题提交给 LLM 生成回答
  • 回答中附带引用来源（指向原始文档和具体段落）
  • 前端展示问答界面：问题输入 → 加载状态 → 回答 + 引用卡片
  • 支持引用来源点击跳转或展开查看原始段落

技术选型建议

  • LLM：初期可使用 OpenAI GPT 系列，通过配置支持其他模型（Claude、本地部署等）
  • Prompt 工程：系统提示词明确要求基于提供的参考资料回答，并标注引用编号
  • 前端交互：从当前单行输入扩展为完整的对话式界面，支持流式输出和引用展示
  • 设计模型适配器接口：统一 generate(prompt, context) → response 协议

验收标准

  • 用户可输入问题并得到基于知识库内容的回答
  • 回答中的关键结论可追溯到原始文档和段落
  • 无相关文档时有明确的兜底回复
  • 前端流畅展示问答交互和引用来源
