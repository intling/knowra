knowra 文件解析与分块功能探索

本文沉淀“消费已上传原始文件，生成文档记录和可追溯 chunk”的探索结论。它不是业务代码实现，而是进入 OpenSpec proposal 前的方案草图，用来明确能力边界、数据契约、一致性要求和 TDD 起点。

模式：openspec-explore 日期：2026-05-28 当前 OpenSpec：无活跃变更 输入：uploaded_files.status = stored 输出：documents + document_chunks 首批格式：TXT / Markdown / PDF / DOCX / PPT/PPTX 分块大小：BPE tokenizer

1. 探索结论

建议把“文件解析与分块”设计为上传之后、embedding 与检索之前的独立后端能力。它从 uploaded_files.storage_key 读取原始文件，创建 documents 记录，再生成稳定、有序、可追溯的 document_chunks。后续 embedding、检索和 RAG 都只消费这些稳定 chunk，而不是直接消费原始上传文件。

这个阶段的核心价值是建立“资料可被系统理解”的中间层。uploaded_files 只说明文件已经被用户上传并保存；documents 说明系统已经成功解析出可处理文本；document_chunks 则是后续向量化、召回、引用展示和评测的最小稳定单元。

首批确认支持 text/plain、text/markdown、PDF、DOCX 和 PPT/PPTX。由于当前后端依赖尚未包含 PDF、Office 文档解析库和 BPE tokenizer，正式 OpenSpec 变更需要把依赖选择、锁文件更新、失败样例和解析质量边界列为显式任务。OCR 和多模态理解仍建议后置。

2. 当前上下文

项目事实 对本能力的影响
已有 uploaded_files 表，字段包括 owner_user_id、original_filename、content_type、byte_size、storage_key、checksum_sha256、status。 解析入口不应重新上传文件；应通过已持久化的上传记录读取原始文件，并复用用户归属与存储键。
UploadService 使用 LocalFileStorage 将原始文件写入 UPLOAD_STORAGE_DIR 下的 server-generated storage_key。 文档处理服务需要复用或扩展文件读取抽象，避免直接拼接用户输入路径。
上传 API 为 POST /api/uploads，成功后返回 201 和上传元数据。 解析 API 应以 uploaded_file_id 为入口，并由后端校验当前用户归属，不接受客户端提交的 owner_user_id。
后端当前依赖未包含 DOCX/PPT/PDF/OCR 解析库，也未包含 BPE tokenizer。 首批已确认纳入 PDF、DOCX/PPT 和 BPE tokenizer，因此 OpenSpec 需要显式说明新增依赖、可替换接口和验证方式。
项目章程要求 OpenSpec 先行，并且实现类任务必须严格 TDD。 本探索完成后，应创建正式 OpenSpec change，再按 Red-Green-Refactor 实施。

3. 系统位置

接入资料
  └─ uploaded_files              已有：原始文件与上传元数据
      └─ documents               本能力：解析后的文档级记录
          └─ document_chunks      本能力：可检索、可引用的文本片段
              └─ chunk_embeddings 后续：向量表示
                  └─ retrieval    后续：召回与排序
                      └─ RAG       后续：基于引用生成回答

这个边界可以避免后续模块直接依赖上传文件格式。检索只需要知道 chunk id、内容、归属和来源定位；引用展示再沿着 document_chunks -> documents -> uploaded_files 找回原始文件名、文档标题和位置线索。

4. 推荐数据流

1. 选择输入 接收 uploaded_file_id，查询当前用户名下且 status = stored 的上传记录。
2. 读取原文 通过 storage_key 从应用内存储读取 bytes 或 stream，不接受客户端路径。
3. 解析文本 根据 MIME 与扩展名选择 parser，输出 normalized text 与 source map。
4. 生成分块 使用确定性 chunker 生成有序 chunk，记录范围、hash、版本和来源定位。
5. 原子入库 在事务中写入 document 与 chunks，失败时不留下可检索半成品。 
uploaded_file(id, owner_user_id, storage_key, content_type, checksum_sha256)
        │
        ▼
FileStorage.open(storage_key)
        │
        ▼
ParserRegistry.resolve(content_type, filename)
        │
        ▼
ParsedDocument(text, metadata, source_map)
        │
        ▼
Chunker.split(text, source_map)
        │
        ▼
documents + document_chunks

5. 能力边界

范围 首批做 首批不做
输入 消费当前用户已上传且成功存储的单个原始文件。 不重新实现上传，不接受任意文件路径，不处理跨用户文件。
解析 抽象 parser 接口，首批支持 TXT、Markdown、PDF、DOCX、PPT/PPTX，输出规范化文本和来源映射。 不做 OCR、图片理解、复杂表格结构恢复、PDF 复杂版面重建、幻灯片视觉版面重建。
分块 使用 BPE tokenizer 计算 chunk 大小和重叠窗口，保存顺序、字符范围、token 数、hash、source locator。 不把 BPE tokenizer 与某个 embedding 模型强绑定；tokenizer 名称、版本和参数必须可记录、可替换。
持久化 新增 documents 和 document_chunks 数据模型与迁移。 不把 chunk 只存在内存或临时文件里。
后续衔接 为 embedding、检索和引用展示保留稳定主键、版本和来源定位。 不实现向量索引、召回 API、重排、RAG 回答。

6. 数据模型草案

documents

字段 类型倾向 说明
id UUID 文档记录主键。
owner_user_id UUID FK 冗余保存用户归属，来自上传记录，不由客户端提交。
uploaded_file_id UUID FK unique 关联 uploaded_files.id。首批建议一个上传文件只对应一个当前文档。
title String 默认使用原始文件名去扩展名，后续可由 parser 或用户修正。
source_content_type String nullable 记录解析时使用的内容类型，便于排查 parser 选择。
parser_name / parser_version String 记录解析器身份，支持后续判断是否需要重跑。
chunker_name / chunker_version String 记录分块策略身份，避免策略变更后旧 chunk 语义不明。
tokenizer_name / tokenizer_version String 记录 BPE tokenizer 身份和版本，保证 chunk 大小、重叠窗口和 token 统计可复现。
status String 首批建议 parsed、failed、deleted。
chunk_count Integer 成功分块数量，用于列表展示和一致性校验。
total_chars Integer 规范化文本字符数，用于诊断空解析和异常膨胀。
content_sha256 String 规范化文本 hash，不等同于原始文件 checksum。
metadata_json JSON 页数、语言、标题层级摘要等非契约核心信息。
error_message String nullable 解析失败时保存可诊断信息。
created_at / updated_at DateTime 审计字段。

document_chunks

字段 类型倾向 说明
id UUID chunk 主键，后续 embedding、检索结果和引用都引用它。
document_id UUID FK 关联 documents.id。
owner_user_id UUID FK 冗余用户归属，支持后续按用户过滤检索。
chunk_index Integer 文档内从 0 开始的稳定顺序，建议与 document_id 做唯一约束。
content Text 用于 embedding 和检索展示的规范化片段文本。
content_sha256 String chunk 内容 hash，用于去重、调试和重跑比对。
char_start / char_end Integer chunk 在文档规范化文本中的半开区间 [start, end)。
token_count Integer 由选定 BPE tokenizer 计算得到，作为 chunk 大小控制和后续模型上下文预算的基础字段。
source_locator_json JSON 页码、段落序号、标题路径、原始块 id 等来源定位信息。
metadata_json JSON 块类型、标题、语言、表格标记等可扩展信息。
created_at / updated_at DateTime 审计字段。
首批不建议把 embedding 状态放进 document_chunks 主表。后续可新增 chunk_embeddings，用 chunk_id + embedding_model + embedding_version 管理不同向量模型的结果。

7. 解析器策略

类型 优先级 建议方式 注意点
text/plain 必须 读取文本，UTF-8 优先；解码失败要显式失败。 不要静默吞掉乱码；失败原因应进入 error_message。
text/markdown 必须 保留 Markdown 文本，并提取标题路径作为 chunk 元数据。 首批不需要渲染 HTML，避免不必要的结构转换。
application/pdf 必须 引入解析库后逐页抽取文本，source locator 至少记录页码。 首批支持具备文本层的 PDF；扫描版 PDF 可能无法抽出文本，应生成 failed 文档并给出原因，首批不做 OCR。
application/vnd.openxmlformats-officedocument.wordprocessingml.document 必须 通过 DOCX 解析库按标题、段落和表格文本抽取内容，source locator 至少记录段落序号或结构路径。 批注、修订痕迹和精确页码可以后置，但不能静默丢弃主要正文。
application/vnd.openxmlformats-officedocument.presentationml.presentation / application/vnd.ms-powerpoint 必须 通过 PPT/PPTX 解析库按幻灯片顺序抽取文本框、标题和备注文本，source locator 至少记录 slide index。 首批只要求文本抽取，不恢复视觉布局、图像内容、动画和复杂图表语义。
parser registry 应返回“明确不支持”的错误，而不是落到默认二进制文本解析。对知识库来说，乱码 chunk 比失败更危险，因为它会污染后续索引和问答引用。PDF、DOCX 与 PPT/PPTX 的首批支持也应建立在专用解析器上，而不是把二进制文件当作普通文本兜底处理。

8. 分块策略

首批推荐

使用确定性的递归文本分块：优先按 Markdown 标题、PDF 页码与段落、Office 文档结构、幻灯片边界、空行、段落、句子边界切分；超过阈值时再按 BPE token 窗口切分。默认目标大小可先采用 800-1200 BPE tokens，重叠 100-200 BPE tokens。

BPE 约束

chunk 稳定性来自同一份规范化文本、同一个 chunker 版本、同一个 BPE tokenizer 版本、同一组参数和确定性边界规则。任一条件变化，都应视作新一轮处理结果，而不是悄悄覆盖旧语义。 

normalized_text:
0         420       860       1290
|----------|----------|----------|

chunks:
chunk 0: [0, 520)      source: page 1, heading "导论"
chunk 1: [420, 940)    source: page 1-2, heading "背景"
chunk 2: [840, 1290)   source: page 2, heading "方法"

overlap:
chunk 0 / 1 overlap = [420, 520)
chunk 1 / 2 overlap = [840, 940)

对 RAG 来说，chunk 不是随便切开的字符串，而是引用能力的基础。每个 chunk 至少要能回答三个问题：它属于谁、来自哪份文档、在原文或规范化文本中的位置在哪里。

9. API 草案

POST /api/documents 从已上传文件创建文档和 chunks。 请求体包含 uploaded_file_id；后端校验归属和 stored 状态。
GET /api/documents/{id} 读取文档记录。 返回解析状态、chunk 数量、标题、来源上传文件信息。

10. 状态与失败语义

场景 推荐行为 原因
上传文件不存在或不属于当前用户 返回 404 避免泄露其他用户资源是否存在。
上传文件未成功存储 返回 409 只有 stored 原始文件能进入解析。
内容类型不支持 返回 415，并创建或返回 failed 文档元数据 failed 文档需要展示在前端资料列表中，用户必须能看到失败原因并理解下一步。
原始文件缺失 返回 500，并记录 failed 文档 数据库记录与文件存储不一致，需要在资料列表和服务日志中都可诊断。
解析结果为空 创建 failed 文档并记录原因 扫描版 PDF 或二进制误传时，用户需要明确反馈。
chunk 写入中途失败 事务回滚，不暴露半套 chunks 后续检索不能消费不完整文档。
最重要的一致性要求是：不能出现 documents.status = parsed 但 chunk 缺失或不完整的状态。成功文档必须拥有与 chunk_count 一致的 chunk 记录。
资料列表必须包含 failed 文档。列表项至少展示标题、来源文件名、状态、失败原因、创建时间和可重试入口；但 failed 文档不能被 embedding、检索或 RAG 流程消费。

11. TDD 起点

真正进入实现前，应先创建 OpenSpec 变更，再严格按失败测试驱动最小实现。推荐测试顺序如下：

  1. 迁移测试：迁移后存在 documents 和 document_chunks 表、索引和外键。
  2. 服务成功测试：给定当前用户的 TXT/Markdown/PDF/DOCX/PPT 上传记录，生成 document 和有序 chunks。
  3. 来源测试：每个 chunk 包含 document_id、owner_user_id、chunk_index、字符范围和 source locator。
  4. BPE 测试：chunk 大小和重叠窗口由 BPE tokenizer 控制，并写入 token_count、tokenizer_name、tokenizer_version。
  5. 权限测试：不能处理不属于当前用户的 uploaded_file_id。
  6. 失败测试：不支持类型、缺失原始文件、空解析结果不会产生可检索的成功 chunks。
  7. 重复处理测试：同一 uploaded_file_id 再次处理返回 409 Conflict，响应体包含已有文档元数据。
  8. 列表测试：GET /api/documents 同时返回 parsed 和 failed 文档，failed 项展示失败原因且不会进入检索消费。
  9. API 测试：POST /api/documents 返回约定响应，并覆盖 404、409、415、500 等分支。

12. OpenSpec 建议

这个探索已经足够进入独立变更。建议 change id 使用 add-document-parsing-chunking，能力名使用 document-processing 或 document-parsing-chunking，与已归档的 file-upload-storage 形成连续链路。

建议目标

  • 消费已上传原始文件，创建文档记录。
  • 首批支持 TXT、Markdown、PDF、DOCX、PPT/PPTX 的文本解析。
  • 基于 BPE tokenizer 将解析后的文本切分为稳定、可追溯 chunks。
  • 确保 chunk 可追溯到当前用户、上传文件、文档和源位置。
  • 重复处理同一 uploaded_file_id 时返回 409 Conflict，并携带已有文档元数据。
  • 前端资料列表展示 parsed 与 failed 文档，failed 文档展示失败原因。
  • 为后续 embedding、检索和 RAG 引用保留稳定主键与版本信息。

建议非目标

  • 不实现 embedding、pgvector 写入、检索、重排或 RAG。
  • 不实现 OCR、图片解析、PDF 复杂版面重建、复杂表格恢复、幻灯片视觉布局恢复或多模态理解。
  • 不实现后台任务队列；如需异步化，应单独设计任务状态和重试机制。

建议任务拆分

  1. 补充 OpenSpec proposal、design、tasks 和规格增量。
  2. 新增数据模型与 Alembic migration。
  3. 选择并接入 PDF、DOCX、PPT/PPTX 解析依赖，以及 BPE tokenizer 依赖。
  4. 实现 parser 接口和 TXT/Markdown/PDF/DOCX/PPT 解析器。
  5. 实现基于 BPE tokenizer 的确定性 chunker 与 source locator 结构。
  6. 实现文档处理服务和 POST /api/documents API。
  7. 实现 409 Conflict 重复处理响应，并返回已有文档元数据。
  8. 实现资料列表中的 failed 文档展示契约。
  9. 同步 README / backend README 中的处理流程和验证命令。

13. 已确认的产品约束

  • 重复处理同一 uploaded_file_id 时，返回 409 Conflict，响应体携带已有文档元数据。
  • 首批支持 TXT、Markdown、PDF、DOCX、PPT/PPTX；OCR 和多模态理解后置。
  • failed 文档需要展示在前端资料列表中，并展示可理解的失败原因。
  • chunk 大小和重叠窗口引入 BPE tokenizer 依赖，不再使用字符近似作为首批正式契约。
推荐下一步是创建 OpenSpec 变更 add-document-parsing-chunking，把本文中的模型、API、状态、一致性、PDF/Office 解析、BPE tokenizer 和 TDD 场景转成正式 artifacts。确认后再退出探索模式进入实现，并按项目章程执行 Red-Green-Refactor。
