knowra 文档分块功能方案（Docling Chunking）

本文沉淀"在现有文档解析能力之后，新增基于 Docling 原生分块器的文档分块能力"的方案。范围覆盖分块作业模型、HybridChunker 适配、分块结果存储、分块作业 API、与解析结果的关系设计，以及后续 embedding 和检索的扩展边界；不在本文中实现代码。

阶段：方案探索 日期：2026-06-10 推荐变更：add-document-chunking-docling 分块核心：Docling HybridChunker + Qwen2-7B tokenizer 分块策略：解析即分块，内存直传 DoclingDocument 禁止路径：从 docling.json 还原后分块 存储策略：≤2KB 入库，>2KB 走文件 前置能力：document-parsing 已归档

1. 方案结论

推荐把"文档分块"设计为解析流程的内嵌步骤：解析 API 负责产出 DoclingDocument，并在同一后台任务中直接将内存中的 DoclingDocument 对象传入分块适配器，无需序列化/还原中间步骤。分块适配器调用 Docling 原生 HybridChunker 将文档分割为带结构元数据的 token 感知块，再持久化分块结果供后续 embedding、检索和引用定位消费。首版使用与解析相同的 BackgroundTasks 调度器，解析成功后自动在同一任务上下文中执行分块，通过 dispatcher 边界保留后续替换为独立 worker + 队列的空间。

分块的核心价值是：为语义检索和 RAG 问答提供粒度可控、结构可追溯、token 长度可感知的文本单元。不是简单地把原文切成等长片段，而是沿着文档的自然结构边界（标题、段落、表格、列表）切割，并在 token 超限时沿结构内部进一步拆分，在欠限时沿相同标题上下文合并。

首个版本的成功标准不是"分块后立刻可以语义检索"，而是"文档被可靠分块为可追溯、可索引、可供后续 embedding 消费的结构化文本单元"。

分块触发策略：解析成功后在同一后台任务中自动执行分块，DoclingDocument 对象直接在内存中传递给分块适配器，无需经过序列化/还原。前置条件：只有解析运行仍持有内存中的 DoclingDocument 时才允许执行分块；如果系统只剩已持久化的 parsed_documents、document_segments 或 docling.json，不得从这些产物还原后启动首次分块。不再需要"手动触发分块"API：首次分块始终在解析成功后自动执行；如果需要重新分块（参数变更），使用 /rechunk API 触发重新解析原始文件并立即分块。重新分块策略：参数变更后重新分块时，必须重新读取原始上传文件并重新解析以获得新的内存 DoclingDocument；旧分块作业标记为 superseded 状态，其关联的 chunk 结果逻辑上不再活跃但保留在数据库中不主动删除；新作业使用新配置独立产出 chunk 集合。查询 chunk 结果时仅返回最新（非 superseded）作业的 chunk。Segment 保留原则：document_segments 表的内容在分块过程中保持不变 — 分块不会修改或合并已有 segment；source_segment_indices 只有在 chunk 与 segment 来自同一次解析运行、映射可靠时才填充，否则保持为空并依赖页码、标题路径和 Docling 元数据追溯来源。产物存储策略：文本内容 ≤2KB 直接入库，>2KB 写入文件存储并通过 storage_key 引用，text/text_storage_key 和 contextualized_text/contextualized_text_storage_key 各自互斥填充。

2. 现有基础与约束

项目现状 对分块方案的影响
已有 parsed_documents 表，保存解析产物 storage_key、标题、页数和元数据，关联 uploaded_file_id 和 parse_job_id。 parsed_documents 只作为持久化关系锚点和查询入口，不能作为运行时分块输入。首次分块必须在解析任务内使用仍在内存中的 DoclingDocument；分块结果关联到本次解析生成的 parsed_document_id。如果只剩已有解析产物，需要通过 /rechunk 重新解析原始上传文件后再分块。

已有 document_parse_jobs 表和 BackgroundTasksParseJobDispatcher，用于异步解析调度。 分块作为解析任务的后续步骤在同一后台任务中执行，不单独创建分块调度器。解析任务成功后直接在内存中调用分块适配器，分块状态写入 document_chunk_jobs 表。
已安装 docling>=2.97.0，项目依赖已包含 DoclingDocument、DocumentConverter 等核心能力。 HybridChunker 在 docling 和 docling-core 包中均可用，但 HuggingFace tokenizer 分块需要 docling-core[chunking] extra 或直接使用 docling 包中的 HybridChunker。需要确认依赖是否已满足。
项目配置使用 Pydantic Settings + .env 管理，数据库使用 SQLModel + PostgreSQL。 分块配置项（max_tokens、merge_peers、tokenizer 模型等）应加入 Settings 类。分块结果表需要 Alembic migration。

3. Docling 分块能力匹配

Docling 提供三种原生分块器，从 DoclingDocument 直接产出分块结果，无需先导出为 Markdown 再手动切分：

✅ HybridChunker（推荐）

原理：先沿文档层级结构做 HierarchicalChunker 分块，再根据用户指定 tokenizer 的 token 限制做感知拆分合并。

优点：同时尊重文档结构和 token 长度限制；超大块自动拆分、欠小块自动合并（同标题上下文下）；contextualize() 方法为每个块补充层级标题上下文，适合 embedding。

缺点：需要指定 tokenizer 或 max_tokens；tokenizer 依赖增加安装体积（HuggingFace tokenizer 或 tiktoken）。

适用场景：RAG 向量索引 — 需要控制 token 长度且保留文档结构。

HierarchicalChunker

原理：为每个检测到的文档元素生成一个块，默认合并列表项。不控制 token 长度。

优点：完全保留结构，无 tokenizer 依赖。

缺点：无法控制块大小；短元素（如单行标题）独立成块对 embedding 不友好；长段落可能超长。

适用场景：文档结构导出、结构可视化 — 不适合直接用于 embedding。

推荐使用 HybridChunker 作为首版分块器，因为它同时满足两个核心约束：(1) 保留文档结构边界（标题、段落、表格），(2) 控制 token 长度以适配 embedding 模型。同时使用 contextualize() 方法为 embedding 输入补充层级标题上下文。HierarchicalChunker 的角色是在 HybridChunker 内部使用，不需要单独暴露。

HybridChunker 关键参数

参数 说明 knowra 推荐值
max_tokens 每个块的最大 token 数量。oversized 块会被拆分。 默认 512，可配置。需与 embedding 模型的 max input tokens 对齐。
tokenizer 用于 token 计数的分词器。可以是 HuggingFace tokenizer 或 tiktoken。 首版使用 HybridChunker(max_tokens=512, tokenizer=AutoTokenizer.from_pretrained("Qwen/Qwen2-7B"))，以 Qwen2-7B 模型的分词器计算 token 数；后续可配置指定其他 embedding 模型的 tokenizer。
merge_peers 是否合并同标题下欠大小的连续块。默认 True。 默认 True，通过配置项开放控制。
repeat_table_header 跨块表格是否重复表头。默认 True。 默认 True，保持表头上下文。

HybridChunker 输出结构

HybridChunker.chunk(dl_doc) 返回 Iterator[BaseChunk]，每个 chunk 包含：

{
  "text": "块的纯文本内容",
  "meta": {
    "headings": ["第 2 章", "语义检索"],   // 层级标题路径
    "captions": [...],                      // 图表标题
    "page_numbers": [3, 4],                // 所在页码
    "origin": {
      "page_no": 3,
      "doc_reference": "..."               // Docling 内部引用
    }
  }
}

HybridChunker.contextualize(chunk) 方法会在块文本前拼接层级标题，产生适合 embedding 的富文本。例如：

# contextualize 输出示例
"第 2 章 语义检索

用于引用和后续分块的正文片段..."
分词器选择：首版使用 HuggingFace Qwen/Qwen2-7B 模型的分词器（AutoTokenizer.from_pretrained("Qwen/Qwen2-7B")），与后续 embedding 和检索模型对齐。选择 Qwen2-7B 分词器的理由：(1) 对中文文本的分词质量优于通用英文分词器；(2) Qwen2 系列是 knowra 后续 embedding 模型的候选之一，分词器对齐可避免 token 切分不一致；(3) HuggingFace AutoTokenizer 接口稳定，切换模型只需修改配置。分词器文件约 10MB，首次运行自动下载并缓存。

4. 推荐数据流

1. 解析完成 DoclingDocument 对象在内存中就绪，解析产物已持久化。
2. 创建分块作业 解析成功后自动创建分块作业，写入 document_chunk_jobs。
3. 内存直传分块 DoclingDocument 直接在内存中传递给 HybridChunker，无需序列化/还原。
4. HybridChunker 调用 HybridChunker 产出带元数据的 token 感知块。
5. 保存结果 块文本写入存储，元数据和位置信息入库。 
run_parse_job 执行解析
        |
        +--> parsed_documents + docling.json 持久化
        |
        +--> DoclingDocument (内存中，不落地)
        |         |
        |         v
        |   HybridChunker(max_tokens=N, tokenizer=AutoTokenizer.from_pretrained("Qwen/Qwen2-7B"))
        |         |
        |         +--> document_chunk_jobs
        |         |     running -> succeeded | failed
        |         |
        |         +--> document_chunks
        |               sequence_index, chunk_text, token_count,
        |               heading_path, page_numbers, contextualized_text,
        |               metadata_json
        |
        v
  解析 + 分块在同一后台任务中完成
关键设计点：解析即分块（内存直传） — 分块的输入不是从文件还原的 DoclingDocument，而是解析过程中内存里的同一个对象。解析后台任务成功后，直接在内存中将 DoclingDocument 传递给分块适配器，避免了 pickle/JSON 序列化与还原的可靠性风险和版本兼容性问题。分块不再需要独立的"还原"步骤。重新分块时，由于内存中的 DoclingDocument 已不可用，需要重新解析原始文件以重新获得内存中的文档对象。

核心原则：DoclingChunkerAdapter 是项目中唯一接触 Docling 分块 SDK 的地方；服务层只接收项目内定义的 ChunkedDocumentPayload；路由层只接收请求和返回 schema。数据库模型保存 knowra 的稳定契约，不直接暴露第三方 SDK 内部结构。分块适配器直接接收内存中的 DoclingDocument，不做文件还原。
实现符合性检查：现有解析适配器如果只返回 markdown、text 和 docling_json，而没有把 Docling 转换结果里的 document 对象作为 transient 字段带回 run_parse_job，则仍不符合本方案。需要把解析输出契约调整为"持久化 payload + 内存 DoclingDocument"两部分：持久化部分继续写入 parsed_documents、document_segments 和文件产物；内存对象只在当前后台任务内传给 DoclingChunkerAdapter，不入库、不序列化、不从 docling.json 反向恢复。

字段 类型 说明
id UUID PK 主键。
chunk_job_id UUID FK → document_chunk_jobs.id 关联分块作业。
parsed_document_id UUID FK → parsed_documents.id 关联解析结果。便于直接从 parsed_document 查询所有 chunk。

chunk_type String(64) 块类型：例如 text、table、list、mixed。
source_segment_indices JSON nullable 块来源的 segment 序号列表，与 document_segments.sequence_index 对应。仅在 chunk 与 segment 基于同一次解析运行且映射可靠时填充；否则保持为空。
metadata_json JSON nullable Docling chunk 元数据快照，包含 Docling 内部引用等。仅用于调试、引用辅助和结果对比，不作为重新分块的输入来源。
created_at DateTime(tz) 时间戳。

索引建议

-- document_chunk_jobs
CREATE INDEX ix_chunk_jobs_owner_user_id ON document_chunk_jobs(owner_user_id);
CREATE INDEX ix_chunk_jobs_parsed_document_id ON document_chunk_jobs(parsed_document_id);
CREATE INDEX ix_chunk_jobs_status ON document_chunk_jobs(status);

-- document_chunks
CREATE INDEX ix_chunks_chunk_job_id ON document_chunks(chunk_job_id);
CREATE INDEX ix_chunks_parsed_document_id ON document_chunks(parsed_document_id);
CREATE INDEX ix_chunks_owner_user_id ON document_chunks(owner_user_id);
CREATE INDEX ix_chunks_sequence_index ON document_chunks(parsed_document_id, sequence_index);
为什么不复用 document_segments 表？ Segment 是解析阶段产出的粗粒度结构单元（一个标题、一个段落、一个表格），不控制 token 长度。Chunk 是分块阶段产出的 token 感知可索引单元，可能在结构边界上拆分或合并多个 segment。Segment 在分块过程中始终保持不变，不会被合并或修改。两者的生命周期、粒度和用途不同：
  • Segment 的用途：文档预览、结构可视化、来源追溯。
  • Chunk 的用途：embedding 输入、向量检索、引用定位。
两者在映射可靠时通过 source_segment_indices 字段保持关联 — 一个 chunk 可以跨越多个 segment，但 segment 原始记录始终完整保留。映射不可靠时不要强填旧 segment 索引，应依赖 page_numbers、heading_path 和 metadata_json 做来源辅助。

7. 分块产物存储

分块产物的文本内容策略需要权衡数据库存储和文件系统存储：

✅ 推荐：阈值混合存储（≤2KB 入库，>2KB 走文件）

chunk.text 和 contextualized_text 统一采用阈值策略：文本长度 ≤ 2KB 时直接存入数据库 Text 字段（短文本快速查询和预览），> 2KB 时写入文件存储，数据库只保存 storage_key。

字段设计：text（Text 可空）存 ≤2KB 的原始文本，text_storage_key（String(1024) 可空）存 >2KB 时的文件引用；contextualized_text（Text 可空）和 contextualized_text_storage_key（String(1024) 可空）同理。

存储路径：

storage/chunked/{owner_user_id}/{parsed_document_id}/{chunk_job_id}/
  chunk_0000.ctx.md        # >2KB 的 contextualized 文本
  chunk_0000.txt           # >2KB 的原始文本
  ...
  chunks_meta.json         # 分块元数据快照

优点：短文本直接在数据库中查询、预览和过滤（大多数 chunk < 2KB）；长文本不占数据库容量；统一阈值逻辑清晰。

纯文件存储

所有文本内容都写入文件，数据库只存 storage_key。

优点：数据库最瘦。

缺点：简单文本预览和搜索需要读文件；增加了存储服务复杂度；与 segment 设计不一致。

推荐阈值混合存储：文本 ≤2KB 入数据库 Text 字段，>2KB 写入文件存储并通过 storage_key 引用。text/text_storage_key 和 contextualized_text/contextualized_text_storage_key 各自一对，互斥填充（一个有值另一个为空）。metadata_json 保留 Docling chunk 原始元数据快照，便于调试、结果对比和引用辅助，但不能作为重分块输入。此策略与 document_segments 的文本存储方式保持一致。

8. DoclingChunkerAdapter 设计

分块适配器是项目中唯一接触 Docling 分块 SDK 的模块。核心流程：

DoclingChunkerAdapter
  ├─ __init__(max_tokens, merge_peers, tokenizer_model="Qwen/Qwen2-7B", ...)
  └─ chunk(dl_doc: DoclingDocument) -> ChunkedDocumentPayload
       ├─ HybridChunker(max_tokens=N, tokenizer=AutoTokenizer.from_pretrained("Qwen/Qwen2-7B"), merge_peers=True)
       ├─ for chunk in chunker.chunk(dl_doc):
       │    ├─ 提取 .text
       │    ├─ 提取 .meta.headings, .meta.page_numbers, .meta.captions
       │    ├─ chunker.contextualize(chunk) → 富文本
       │    └─ 构建 ChunkPayload
       └─ return ChunkedDocumentPayload(chunks=[...], total_token_count=...)

解析输出契约调整

为支持内存直传，解析适配器不能只返回可持久化的 JSON/Markdown/Text 产物，还必须把当前 Docling 转换结果中的 DoclingDocument 作为 transient 运行时对象交还给解析任务。该字段只用于当前进程内调用分块适配器，不参与 JSON 序列化、文件存储或 API 返回。 

@dataclass(frozen=True)
class ParsedDocumentPayload:
    markdown: str
    text: str
    docling_json: dict[str, Any]
    docling_document: Any | None = field(default=None, repr=False, compare=False)
    title: str | None = None
    page_count: int | None = None
    metadata: dict[str, Any] = field(default_factory=dict)
    segments: list[ParsedSegmentPayload] = field(default_factory=list)
实现要求：DoclingParserAdapter._normalize_docling_result() 应从 result.document 取出同一个 DoclingDocument 并放入 transient 字段；save_parse_result() 只保存可持久化字段；run_parse_job 在保存解析结果后、后台任务结束前立即调用 DoclingChunkerAdapter.chunk(payload.docling_document)。如果某种格式当前无法产生 DoclingDocument，首版应明确跳过 Docling 原生分块或将该格式改走 Docling converter，不能退回到从 docling_json 还原。
内存直传设计：分块适配器直接接收内存中的 DoclingDocument 对象，不再包含 restore_docling_document() 方法。DoclingDocument 由解析流程在内存中直接传入，无需从 pickle 或 JSON 文件还原。这消除了序列化/还原的可靠性风险和版本兼容性问题。重新分块时需要重新解析原始文件以获得内存中的 DoclingDocument。

适配器输入输出契约

@dataclass(frozen=True)
class ChunkPayload:
    sequence_index: int
    text: str | None          # ≤2KB 时有值，>2KB 时为 None，走 text_storage_key
    text_storage_key: str | None  # >2KB 时有值，≤2KB 时为 None
    contextualized_text: str | None     # ≤2KB 时有值，>2KB 时为 None
    contextualized_text_storage_key: str | None  # >2KB 时有值
    token_count: int | None
    heading_path: list[str] | None
    page_numbers: list[int] | None
    chunk_type: str              # "text" | "table" | "list" | "mixed"
    source_segment_indices: list[int] | None
    metadata: dict[str, Any]

@dataclass(frozen=True)
class ChunkedDocumentPayload:
    chunks: list[ChunkPayload]
    total_token_count: int | None
    chunker_name: str
    chunker_version: str | None
    chunk_config: dict[str, Any]
不再需要还原策略：采用内存直传方案后，分块适配器不再需要从文件还原 DoclingDocument。DoclingDocument 由解析流程在内存中直接传入分块适配器，这从根本上规避了 pickle 序列化的跨版本兼容性风险和 JSON 反序列化的稳定性问题。trade-off：重新分块时需要重新解析原始文件（因为内存中的 DoclingDocument 已不可用），这是为了消除还原风险的合理代价。

9. API 草案

接口 用途 响应语义
POST /api/parsed-documents/{parsed_document_id}/rechunk 以当前解析结果作为用户入口，找到其关联的原始上传文件，使用新参数重新解析原始文件并立即分块。允许指定 max_tokens、merge_peers 等参数覆盖默认值。不得读取 docling.json 还原 DoclingDocument。旧分块作业及其结果标记为 superseded 状态，新作业独立产生新的 chunk 集合。 返回 202 Accepted 和新分块作业；存在运行中作业返回 409；原始上传文件不存在时返回可诊断错误。
GET /api/document-chunk-jobs/{job_id} 查询分块作业状态。 只允许当前用户读取自己的作业。
GET /api/parsed-documents/{parsed_document_id}/chunks 分页查询分块结果。 仅返回最新非 superseded 分块作业的结果，按 sequence_index 返回，支持分页。
GET /api/document-chunks/{chunk_id} 查询单个 chunk 详情（含 contextualized_text）。 返回块文本、元数据、页码、标题路径和 contextualized 文本。

分块触发时机

方案 A：解析成功后内存直传自动分块 ✅ 首版采用

解析后台任务成功后，直接在内存中将 DoclingDocument 传递给分块适配器。不需要独立的分块触发 API。

优点：消除 pickle/JSON 还原风险；用户无需手动操作，端到端体验流畅；架构简单。

缺点：分块参数（如 max_tokens）在首次分块时使用默认配置；分块和解析耦合在同一任务中。

方案 B：参数变更后重新分块

需要重新分块时，通过 POST /rechunk 触发重新解析原始文件 + 重新分块流程。旧作业标记为 superseded，新作业使用新配置独立产出 chunk 集合。

优点：参数变更后可重新分块。

缺点：重新分块需要重新解析原始文件（因为 DoclingDocument 不持久化到文件）。

分块触发策略：(1) 解析成功后在同一后台任务中自动执行分块（内存直传 DoclingDocument，使用默认配置）；(2) 不再提供独立的 POST /chunk API — 首次分块始终由解析流程自动触发；(3) 参数变更后可通过 POST /rechunk API 重新分块（内部会重新解析原始上传文件以获得内存中的 DoclingDocument，不读取 docling.json），旧作业标记为 superseded 并保留、新作业独立产出 chunk 集合。仅解析成功（status=succeeded）的文档可查询分块结果。

10. 作业执行策略

分块在解析后台任务内自动执行，不单独调度：

run_parse_job 执行流程：
  1. 解析原始文件 → DoclingDocument
  2. 构建 ParsedDocumentPayload，其中 docling_document 为 transient 内存字段
  3. 持久化解析产物（docling.json, content.md 等），但不持久化 docling_document
  4. 更新 parse_job 状态为 succeeded
  5. 如果 DOCUMENT_CHUNKING_ENABLED=true：
     a. 创建 document_chunk_jobs 行（status=running）
     b. 校验 payload.docling_document 存在；不存在则将 chunk_job 标记为 failed
     c. 在同一任务中调用 DoclingChunkerAdapter.chunk(payload.docling_document)
     d. 保存 chunk 产物（≤2KB 入库，>2KB 走文件存储）
     e. 更新 chunk_job 状态为 succeeded/failed

重新分块（POST /rechunk）：
  1. 校验 parsed_document 归属，并通过 uploaded_file_id 找到原始上传文件
  2. 创建 document_chunk_jobs 行（status=queued/running）
  3. 标记旧分块作业为 superseded
  4. 重新解析原始上传文件 → 获得新的内存 DoclingDocument（不读取 docling.json）
  5. 调用 DoclingChunkerAdapter.chunk(payload.docling_document) 使用新配置
  6. 保存 chunk 产物
  7. 更新 chunk_job 状态
核心变更：首次分块不再是可从持久化解析产物启动的独立作业，而是解析任务的后续步骤。DoclingDocument 在内存中直接传递，不需要独立的 dispatch 机制和文件还原步骤。只有重新分块（/rechunk）需要重新解析原始文件；重新解析的目的只是重新获得内存文档对象，不允许从旧 docling.json 还原。分块状态仍然持久化到 document_chunk_jobs 表以便查询和追溯。

14. MVP 范围

纳入 MVP

  • 新增 OpenSpec 变更 add-document-chunking-docling。
  • 分块作业表和分块结果表（含阈值混合存储字段和 superseded 状态）。
  • 调整解析适配器输出契约，保留 transient DoclingDocument 供同一后台任务内分块使用；该对象不入库、不写文件、不从 docling.json 还原。
  • Docling HybridChunker 适配器，直接接收内存中的 DoclingDocument 对象执行分块（无需从文件还原）。
  • 解析成功后自动在同一后台任务中执行分块（内存直传 DoclingDocument，使用默认配置）。
  • 参数变更后重新分块（POST /rechunk，触发重新解析原始上传文件 + 重新分块，旧作业标记 superseded）。
  • 分块结果的阈值混合存储（≤2KB 入库，>2KB 走文件），document_segments 在分块过程中保持不变；source_segment_indices 仅在同源映射可靠时填充。
  • 分块作业状态查询、结果分页查询、重新分块 API。
  • 权限校验：只能查看自己的分块结果。

暂不纳入 MVP

  • 向量 embedding 和 pgvector 索引。
  • 语义检索和 RAG 问答。
  • 自定义分块参数的 API（如用户指定 max_tokens 或 tokenizer）— 注：首版通过 rechunk 端点支持参数变更后重分块，但不支持在首次自动分块时自定义参数。
  • 多种分块策略切换（如 HierarchicalChunker 或自定义分块器）。
  • 分块结果向量化或向量化存储。
  • 引用生成和来源展示。
  • 独立 worker + 队列的生产级部署。
  • 跨解析运行的分块结果与旧 segment 自动对齐算法（首版不做不可靠强对齐）。

16. 风险与开放问题

风险 影响 缓解方式
Docling JSON 还原 DoclingDocument 不可靠 分块无法从已有产物启动，需要重新解析。 已采用内存直传方案彻底规避：分块在解析后台任务内执行，DoclingDocument 直接在内存中传递给分块适配器，不再需要从 pickle 或 JSON 文件还原。这从根本上消除了还原不可靠的风险。trade-off：重新分块时需要重新解析原始文件以获得内存中的 DoclingDocument。
解析适配器输出丢失 DoclingDocument 即使方案写成内存直传，实现时也可能因为只拿到 docling_json 而被迫回到不可靠还原路径。 必须调整解析输出契约：解析结果需要同时包含可持久化 payload 和 transient DoclingDocument。run_parse_job 在保存解析结果后立即使用该内存对象执行分块；如果对象不存在，分块作业失败并提示格式或解析适配器不支持 Docling 原生分块，禁止静默回退到 JSON 还原。
HybridChunker 依赖 tokenizer 增加 bundle 体积 如果使用 HuggingFace tokenizer，需要下载模型权重。 首版使用 HuggingFace Qwen/Qwen2-7B 分词器（transformers.AutoTokenizer），分词器文件约 10MB，首次运行自动从 HuggingFace Hub 下载并缓存；后续可配置指定其他 embedding 模型的 tokenizer。
分块作业失败后无法单独重试 分块失败后需要重新解析才能重试。 分块在解析任务内执行，失败时需要通过 /rechunk 触发完整的重新读取原始上传文件、重新解析 + 重新分块流程。分块状态仍然持久化到 document_chunk_jobs 表，便于诊断和追溯。
参数变更后重新分块 允许用户使用新参数（如不同 max_tokens）重新分块。 通过 /rechunk 触发重新解析原始上传文件 + 重新分块完整流程，明确不读取旧 docling.json。旧作业标记为 superseded，新作业使用新配置独立产出 chunk 集合。
大文档分块耗时不可控 与解析类似，大文档可能占用大量 CPU 和内存。 设置 max_tokens 限制单块大小；分块作业状态持久化；失败可重试。
BackgroundTasks 不具备生产级可靠性 与解析作业相同的风险。 分块在解析任务内执行，风险与解析相同。后续与解析一起替换为独立 worker + 队列。
chunk.text 入库可能增加数据库存储 大文档的分块文本累计可能较大。 采用 ≤2KB 入库、>2KB 走文件的阈值策略：大多数 chunk 短文本直接入库方便查询，长文本走文件存储避免膨胀数据库。text/text_storage_key 和 contextualized_text/contextualized_text_storage_key 各自互斥填充，与 document_segments 的文本存储策略保持一致。
参数变更后重新分块需要管理旧结果 用户更换 embedding 模型或调整 max_tokens 后，旧分块不再适用；旧 chunk 数据若不清理会累积占用存储。 旧作业标记为 superseded 状态，新作业独立产出 chunk 集合。查询 chunk 结果时仅返回最新（非 superseded）作业的 chunk。旧结果不主动删除，保留用于回溯和对比；后续可提供按 parsed_document 维度的清理 API，由用户或运维触发。

17. OpenSpec 建议

建议下一步创建 OpenSpec 变更 add-document-chunking-docling，新增能力规格 document-chunking。该变更应聚焦"在解析运行内从内存 DoclingDocument 生成 token 感知的结构化分块"，不要同时承诺 embedding、检索或 RAG。

建议 requirements

  • 系统 SHALL 在解析后台任务成功后为当前用户创建文档分块作业，并持久化作业状态以便查询。
  • 系统 SHALL 调整解析输出契约，使解析结果同时包含可持久化 payload 和 transient DoclingDocument；transient 对象只允许在当前后台任务内使用，不入库、不写文件、不通过 API 返回。
  • 系统 SHALL 在解析后台任务成功后自动使用默认分块配置在同一任务中执行分块，将内存中的 DoclingDocument 直接传递给分块适配器。
  • 系统 SHALL NOT 提供独立的 POST /chunk API；首次分块始终由解析流程自动触发。
  • 系统 SHALL 支持用户通过 /rechunk API 使用自定义参数重新分块；重新分块必须重新读取原始上传文件并执行完整解析 + 分块流程，旧作业标记为 superseded，新作业独立产出 chunk 集合。
  • 系统 SHALL 使用 Docling HybridChunker（tokenizer 为 Qwen/Qwen2-7B）直接从内存中的 DoclingDocument 执行分块，不再从文件还原文档对象。
  • 系统 SHALL NOT 从 docling.json、pickle 或其他已落地解析产物还原 DoclingDocument 后执行分块。
  • 系统 SHALL NOT 在分块适配器外部暴露 Docling SDK 内部类型。
  • 系统 SHALL 持久化分块作业状态、分块配置快照、分块器版本和错误信息。
  • 系统 SHALL 在分块作业标记为 superseded 时保留旧 chunk 结果不主动删除，查询时仅返回最新（非 superseded）作业的 chunk。
  • 系统 SHALL 持久化每个分块的文本、token 计数、层级标题、页码、元数据，并在同源映射可靠时持久化来源 segment 索引。
  • 系统 SHALL NOT 在分块过程中修改或合并已有的 document_segments 记录；segment 始终保留完整。
  • 系统 SHALL 对分块产物采用阈值混合存储：文本 ≤2KB 入数据库，>2KB 写入文件存储并通过 storage_key 引用。text/text_storage_key 和 contextualized_text/contextualized_text_storage_key 各自互斥填充。
  • 系统 SHALL 在首版将分块逻辑嵌入解析后台任务中执行，并保留后续替换为独立 worker + 队列的服务边界。
  • 系统 SHALL 防止用户读取不属于自己的分块结果。
  • 系统 SHALL NOT 在首版实现 embedding、pgvector 索引、语义检索或 RAG 问答。

建议 tasks

  1. 创建 OpenSpec proposal、design、tasks 和 document-chunking 规格增量。
  2. Spike 验证：HybridChunker 调用方式；验证从内存中直接传入 DoclingDocument 对象执行分块的可行性。
