# knowra 项目章程

## 1. 项目核心定义

**knowra（知寻）** 是一个围绕用户私有资料构建的 AI 知识助手。它面向个人用户和团队用户，帮助用户将课程讲义、论文、笔记、项目文档等资料整理为可检索、可追溯、可问答的知识库。

knowra 不是通用聊天机器人，也不是普通文件管理器。它的核心价值是让用户能用自然语言访问自己的资料，并让 AI 回答尽量基于真实文档内容生成，同时提供可追溯的引用来源。

## 2. 产品目标

knowra 关注以下产品目标：

- **私有知识沉淀**：把用户上传或接入的资料转化为结构化、可检索的个人或团队知识库。
- **语义检索**：理解用户问题的含义，而不只依赖关键词匹配。
- **基于资料的问答**：在回答问题时优先使用用户知识库中的内容，减少脱离资料的泛化回答。
- **引用与溯源**：回答中的关键结论应能追溯到原始文档、段落或知识片段。
- **可扩展知识能力**：为 Query Rewrite、HyDE、RAG、知识图谱、检索评测等能力保留清晰扩展空间。

## 3. 核心用户流程

knowra 的核心工作流是：

```text
接入资料 -> 解析内容 -> 分块与索引 -> 用户提问 -> 检索相关片段 -> 生成带引用的回答
```

围绕这条主线，系统需要支撑文档理解、元数据管理、语义索引、检索编排、回答生成和来源展示等能力。具体实施计划、能力优先级和验收要求由 OpenSpec 变更文档管理。

## 4. 系统边界

### 4.1 前端职责

前端负责用户可见的产品体验，包括资料接入、知识库浏览、提问交互、回答展示、引用来源展示、状态反馈和错误提示。

前端子项目的技术选型、目录结构、开发命令、测试要求和端侧工程规范以 `front/AGENTS.md` 为准。

### 4.2 后端职责

后端负责 API 服务、配置管理、数据库访问、文档处理服务、索引与检索编排、模型服务适配、RAG 流程、知识图谱与评测能力的服务端边界。

后端子项目的技术选型、目录结构、开发命令、测试要求和端侧工程规范以 `backend/AGENTS.md` 为准。

### 4.3 跨端契约

前后端通过统一 API 契约协作。接口路径、请求响应结构、错误语义和跨端行为变更，应优先在 OpenSpec 变更中说明，并同步更新相关 README 和端侧章程。

## 5. OpenSpec 治理

本项目使用 OpenSpec 进行 SDD（Spec-Driven Development）管理。

- 新功能、行为变更、接口变更、架构调整，应先通过 OpenSpec 描述背景、设计、任务和验收方式。
- `AGENTS.md` 负责说明稳定的项目定位、协作边界和工程规则；具体实施计划、任务拆分和验收细节由 OpenSpec 管理。
- 实现前应阅读相关 OpenSpec artifacts；如果实现过程中发现设计偏差，应先回到 OpenSpec 更新方案或任务，再继续实现。
- 归档后的 OpenSpec 变更是项目历史决策来源，后续修改应避免与已归档决策冲突；确需调整时应创建新的变更说明原因。

## 6. Superpowers 与 TDD

本项目开发采用 Superpowers 插件作为工程协作流程约束。OpenSpec 负责定义“做什么”和“为什么做”，Superpowers 负责约束“如何探索、计划、实现、验证”。

实现任何新功能、Bug 修复、重构或行为变更时，必须严格遵循 Superpowers 的 TDD 规范：

- **先测试后实现**：没有先写失败测试，不得编写生产代码。
- **先确认 RED**：新增测试必须先运行并确认以预期原因失败。
- **最小 GREEN**：只编写让失败测试通过的最小实现，不夹带未被测试驱动的额外能力。
- **再重构**：只有在测试通过后才能整理结构、命名和重复代码，重构期间必须保持测试通过。
- **循环推进**：每个独立行为都应按 Red-Green-Refactor 循环推进。
- **禁止事后补测替代 TDD**：实现后再补测试不等同于 TDD；如已先写实现，应回退实现并从失败测试重新开始。

允许例外仅限探索性原型、生成代码或纯配置调整等特殊情况。例外必须在开始前明确说明原因，并由项目维护者确认。

完成任何实现类任务前，必须按 Superpowers 的 verification-before-completion 要求运行相应验证命令，并基于实际输出汇报结果。不得在未运行验证的情况下声称功能完成、测试通过或构建正常。

## 7. 全局工程原则

- **先定义契约，再实现细节**：跨模块、跨端或影响用户行为的改动，应先明确目标、输入输出和验收方式。
- **模块化边界清晰**：文档解析、向量化、检索、RAG、知识图谱、评测等能力应保持可替换、可测试的模块边界。
- **配置隔离**：敏感配置不得硬编码，统一通过环境变量和配置模板管理。
- **可验证交付**：代码变更应配套必要测试或验证命令，不能只依赖人工判断。
- **文档同步**：接口、启动方式、目录结构或关键工程约定变化时，应同步更新根目录 README、对应端侧 README 和端侧 `AGENTS.md`。
