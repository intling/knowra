## Context

当前项目已经具备 FastAPI、SQLModel、Alembic、PostgreSQL + pgvector、Vue 3、Pinia 和统一 `/api` 前缀等基础工程骨架，但尚未定义用户实体。后续资料上传、文档解析、知识库归属、问答记录和引用追溯都需要知道资源属于哪个用户。

本变更的设计重点是建立“用户归属数据契约”，不是建立认证系统。认证、注册、登录、权限和团队协作都应作为后续独立变更接入到同一 `users.id` 上，避免当前阶段把未确定的认证形态写入核心模型。

## Goals / Non-Goals

**Goals:**

- 建立稳定的 `users` 表，支持后续资源通过 `owner_user_id` 关联用户。
- 通过 `deleted_at` 支持用户软删除，并让默认用户查询排除软删除记录。
- 提供当前用户解析服务，本阶段返回系统默认用户。
- 提供 `GET /api/users/me`，让前端和后续上传流程获得当前用户上下文。
- 前端封装当前用户 API，使用轻量 user store 表达加载中、成功和失败状态。
- 明确前端不提交普通业务资源的 `owner_user_id`，资源归属由后端根据当前用户解析结果写入。

**Non-Goals:**

- 不实现登录、注册、退出、会话、访问令牌、Cookie Session 或权限中间件。
- 不实现密码哈希、OAuth 身份映射、邮箱验证或用户资料编辑。
- 不实现组织、团队、多租户、角色或资源授权模型。
- 不提供普通用户可见的用户切换器。
- 不修改检索、RAG、embedding、chunking 或引用生成链路。

## Decisions

### Decision 1: 使用单一 `users` 表作为基础用户资料模型

`users` 表保存基础资料与生命周期字段：

- `id`: UUID 主键。
- `display_name`: 用户展示名称。
- `email`: 可空邮箱，非空时唯一，用于后续邮箱登录预留。
- `avatar_url`: 可空头像地址。
- `status`: 用户状态，初始支持 `active` 和 `disabled`。
- `deleted_at`: 可空软删除时间。
- `created_at`: 创建时间。
- `updated_at`: 更新时间。

选择理由：当前阶段只需要资源归属和用户上下文。把密码、OAuth、邮箱验证等认证字段放进 `users` 会提前耦合认证设计，后续扩展成本更高。

替代方案：

- 自增整数主键：实现简单，但跨环境合并和外部身份映射更不稳。当前选择 UUID。
- 把认证字段一起放入 `users`：短期少一张表，但会混合用户资料与认证凭据。当前不采用。

### Decision 2: 通过 `deleted_at` 实现软删除

用户删除只写入 `deleted_at`，不物理删除记录。常规用户查询、当前用户解析和资源归属检查默认只处理 `deleted_at IS NULL` 的用户。

选择理由：knowra 的文件、文档、知识库和问答历史需要可追溯到原始归属。物理删除用户会破坏历史资源的外键与审计语义。

替代方案：

- 使用 `status = deleted` 表示删除：容易和禁用、停用等可恢复状态混淆。当前保留 `status` 表达可恢复状态，用 `deleted_at` 表达删除时间。
- 物理删除用户：会破坏历史资源归属。当前不采用。

### Decision 3: 当前用户解析服务返回默认用户

后端提供统一当前用户解析函数或依赖，例如 `get_current_user()`。本阶段它返回系统默认用户，并保证该用户未软删除且状态为 `active`。

后续上传文件、创建文档、创建知识库等流程必须通过该解析能力获取用户，不能在业务代码中硬编码默认用户 ID。

选择理由：业务代码依赖“当前用户”抽象，而不是依赖“默认用户”实现。后续接入登录注册时，只需替换解析逻辑，不需要重写资源创建流程。

替代方案：

- 在每个业务服务中硬编码默认用户：短期直接，但会扩大后续认证接入改动面。当前不采用。
- 前端显式传入 `owner_user_id`：会让客户端决定资源归属，不适合后续认证安全边界。当前不采用。

### Decision 4: 只开放 `GET /api/users/me`

本阶段仅提供当前用户读取接口。响应使用后端统一 JSON 字段风格，包含：

```json
{
  "id": "018f4d5a-6a4f-7b62-8f71-8c2e32901a9c",
  "display_name": "Default User",
  "email": null,
  "avatar_url": null,
  "status": "active",
  "deleted_at": null,
  "created_at": "2026-05-23T00:00:00Z",
  "updated_at": "2026-05-23T00:00:00Z"
}
```

如果默认用户不存在、被禁用或已软删除，接口返回明确错误，不静默降级为匿名用户。

选择理由：`/users/me` 是认证系统中也常见的稳定契约。当前不暴露 `POST /api/users`，避免形成事实上的注册接口。

替代方案：

- 暴露完整用户 CRUD：缺少管理场景，会扩大范围。当前不采用。
- 不提供用户 API，仅内部使用：前端和上传流程无法获得明确用户上下文。当前不采用。

### Decision 5: 前端只消费当前用户上下文

前端新增：

- `front/src/api/users.ts`: 定义当前用户响应类型与 `getCurrentUser()`。
- `front/src/stores/user.ts`: 保存 `currentUser`、`isLoading`、`error`，提供 `loadCurrentUser()`。
- 页面或应用壳层：展示当前用户名称，展示用户加载失败状态。

前端不新增登录、注册、退出、找回密码或用户切换 UI。上传文件等后续流程不让用户填写 `owner_user_id`。

选择理由：当前用户是全局上下文，适合独立于 app store 管理。前端只展示和消费上下文，资源归属由后端决定。

替代方案：

- 把当前用户混入现有 app store：短期少一个文件，但用户状态会成为多业务模块共享基础状态，独立 store 边界更清楚。
- 做登录注册占位 UI：会制造尚未实现的产品承诺。当前不采用。

## Risks / Trade-offs

- 默认用户初始化缺失 → `GET /api/users/me` 无法返回用户。缓解：服务层提供清晰错误；实现任务中覆盖默认用户不存在分支；必要时在迁移或初始化流程中创建默认用户。
- 业务代码绕过当前用户解析并硬编码默认用户 ID → 后续认证接入困难。缓解：测试覆盖服务层使用当前用户解析；评审时禁止业务层直接写死用户 ID。
- 软删除用户仍被返回 → 已删除用户可能继续关联新资源。缓解：服务层查询默认包含 `deleted_at IS NULL`；测试覆盖软删除用户不可作为当前用户。
- 前端提交 `owner_user_id` → 客户端可伪造资源归属。缓解：规格明确普通业务流程不提交该字段；后续上传接口由后端写入归属。
- 现在不实现认证 → 多用户真实隔离能力有限。缓解：把本变更限定为基础数据支撑，后续用独立 OpenSpec 变更接入认证与权限。

## Migration Plan

1. 新增 Alembic migration 创建 `users` 表、主键、非空约束、邮箱唯一约束和必要索引。
2. 在后端模型中导入 `User`，确保 Alembic 能发现 SQLModel metadata。
3. 通过迁移、初始化服务或测试 fixture 准备一个默认 active 用户。
4. 发布后，后续资源表可通过 `owner_user_id` 外键引用 `users.id`。
5. 回滚时撤销新增接口、服务和前端用户状态，并通过 Alembic downgrade 删除 `users` 表。若已有关联资源表引用用户，应先处理外键依赖后再回滚。

## Open Questions

- 默认用户是否由 Alembic 数据迁移创建，还是由应用启动初始化服务创建，需要在实现时结合现有数据库初始化方式确定。
- 前后端用户字段是否统一使用 snake_case，还是前端 API 层转换为 camelCase，需要在实现前保持一致。
