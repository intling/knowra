## Why

knowra 后续的资料上传、文档处理、知识库归属和问答记录都需要稳定关联到用户，但当前项目还没有用户数据基础。现在先建立不含登录注册的基础用户体系，可以支撑资源归属与追溯，同时避免过早锁定认证形态。

## What Changes

- 新增基础用户能力，提供稳定的 `users` 数据实体，用于后续文件、文档、知识库等资源通过 `owner_user_id` 关联用户。
- 新增用户软删除语义，通过 `deleted_at` 表示用户信息已删除，常规查询和当前用户解析默认排除已删除用户。
- 新增当前用户解析能力，本阶段默认返回系统默认用户，不实现登录、注册、会话、令牌或权限控制。
- 新增 `GET /api/users/me`，用于前端、上传流程和联调获取当前解析出的用户。
- 前端新增当前用户 API 封装和轻量用户状态，不新增登录、注册、退出或用户切换 UI。
- 规定后续资源创建流程应由后端根据当前用户解析结果写入归属，普通前端流程不提交 `owner_user_id`。

## Capabilities

### New Capabilities

- `basic-user-foundation`: 定义基础用户数据、软删除、当前用户解析、`GET /api/users/me` 契约，以及前端当前用户状态接入边界。

### Modified Capabilities

无。

## Impact

- 后端数据模型：新增 `users` 表模型，包含用户展示资料、状态、软删除和时间字段。
- 数据库迁移：新增 Alembic migration 创建 `users` 表、约束和索引。
- 后端服务：新增当前用户解析服务，默认返回未删除、未禁用的默认用户。
- 后端 API：新增 `/api/users/me` 路由和响应 schema。
- 前端 API：新增 `front/src/api/users.ts`，封装当前用户读取。
- 前端状态：新增轻量 user store，表达加载中、成功和失败状态。
- 前端 UX：展示当前用户上下文和用户不可用错误，不引入认证入口。
- 文档：实现时需同步必要 README 或 OpenSpec 说明，明确本阶段不包含登录注册。
