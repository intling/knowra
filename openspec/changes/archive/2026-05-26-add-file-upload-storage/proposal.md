## Why

knowra 的核心流程从“接入资料”开始，但当前项目只有基础用户体系和前端附件选择入口，还没有把用户文件保存到应用内的后端能力。先建立原始文件上传与存储边界，可以让后续文档解析、分块、索引和基于资料的问答有稳定输入来源，并确保文件资源从创建时就可追溯到当前用户。

## What Changes

- 新增单文件上传能力，前端通过 `multipart/form-data` 将文件提交到后端。
- 新增后端上传 API，例如 `POST /api/uploads`，用于保存原始文件并返回上传记录。
- 新增 `uploaded_files` 持久化模型和 Alembic migration，记录文件归属、原始文件名、内容类型、大小、存储键、校验值、状态、错误信息、软删除和时间字段。
- 新增本地应用内文件存储配置和服务边界，使用服务端生成的 `storage_key` 保存原始文件，不把客户端原始文件名直接作为真实路径。
- 上传资源归属沿用现有当前用户解析能力：后端通过 `get_current_user(session)` 写入 `owner_user_id`，前端不提交也不展示可编辑的资源归属用户字段。
- 前端在现有附件选择体验基础上接入上传请求，并展示上传中、成功、失败、用户不可用等反馈。
- 不实现文档解析、分块、embedding、向量索引、检索、RAG、对象存储、登录注册、团队权限或文件下载能力。

## Capabilities

### New Capabilities

- `file-upload-storage`: 定义当前用户文件上传、应用内原始文件存储、上传元数据、API 契约、错误语义和前端上传反馈。

### Modified Capabilities

- 无。现有 `basic-user-foundation` 的资源归属规则保持不变，本变更只消费当前用户解析能力。

## Impact

- 后端 API：新增 `/api/uploads` 相关路由，所有接口挂载在现有 `/api` 前缀下。
- 后端服务：新增上传服务和本地文件存储服务边界，负责大小限制、存储键生成、文件写入、校验值计算和元数据入库。
- 后端配置：新增上传存储目录、单文件大小限制、允许内容类型等配置，并同步 `.env.example`。
- 数据库：新增 `uploaded_files` 表、用户外键、状态/归属/创建时间等索引和 Alembic migration。
- 前端：新增 uploads API 封装，更新现有附件选择/发送流程，表达上传状态和错误反馈。
- 依赖：后端需要支持 `multipart/form-data` 请求解析，若当前依赖未包含对应运行依赖，应在后端依赖中补充并更新锁文件。
- 文档：更新后端 README、前端 README 或根 README 中与上传配置、API 契约和本地存储目录有关的说明。
- 验收信号：有效文件上传后数据库存在归属当前用户的上传记录，应用内存储目录存在对应原始文件；当前用户不可用、空文件、超出大小限制等分支返回明确错误且不产生成功记录。
- 回滚说明：回滚时撤销前端上传入口调用、后端上传路由和服务配置，并通过 Alembic downgrade 删除 `uploaded_files` 表；已写入的本地原始文件需要按存储目录清理策略单独处理。
