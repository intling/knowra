# knowra 项目章程

## 1. 项目简介
knowra 是一个前后端分离的个人知识库 AI 助手项目，旨在构建可运行的基础工程骨架，快速验证前端与后端联通、环境配置、数据库连接与基础健康检查。

项目目标不是完成完整业务功能，而是建立稳定的技术基线，为后续功能扩展（文档上传、语义检索、向量数据库、RAG 问答）奠定清晰可维护的架构。

## 2. 目标与愿景
- 构建可以启动、测试、联调的前后端骨架。
- 确保前端与后端通过统一 API 前缀和代理机制顺畅通信。
- 建立数据库迁移、ORM 模型和基础测试流程。
- 保持代码质量和可读性，为后续迭代降低技术债务。

## 3. 关键成功指标
- 后端服务启动成功并返回 `/api/health` 200。
- 前端成功调用后端健康检查接口并展示结果。
- 数据库连接完成，pgvector 扩展迁移生效。
- 代码库包含基础测试，且本地测试可通过。
- 规范化的开发文档和运行说明存在于 `README.md` 中。

## 4. 项目主导与治理
- 技术决策遵循“简单可验证、先可用再优化”规则，避免过早设计复杂功能。
- 变更流程应通过代码评审、文档更新与测试验证；重大接口或架构调整需先在 `README.md` 或 `AGENTS.md` 中说明。
- 优先解决前后端联调通畅问题。


## 5. 项目内容
### 5.1 包含内容
- 后端：
  - FastAPI 服务与启动入口
  - `/api/health` 健康检查接口
  - PostgreSQL 连接与 `pgvector` 扩展启用迁移
  - SQLModel ORM 基础模型、数据库会话管理
  - 环境配置管理与 `.env.example` 模板
  - 基础测试、日志与代码质量检查
  - 目录：`backend/`

- 前端：
  - Vite + Vue 3 单页应用
  - Pinia 状态管理
  - Vue Router 路由管理
  - 基础 API 客户端、健康检查页面
  - Vite 代理配置，实现前端请求转发到后端
  - CSS 样式与基础页面布局
  - 目录：`front/`

### 5.2 不包含内容
- 文档上传与文件存储实现
- 文档解析、OCR、文本抽取等数据处理
- 向量索引创建与检索逻辑
- RAG 聊天会话、LLM 调用、对话管理
- 用户认证、权限体系、安全策略
- 完整业务流程与生产级部署流水线

## 6. 技术架构与标准
### 6.1 后端技术栈
- Python 3.14.5
- FastAPI
- uvicorn（开发 server）
- gunicorn（生产部署可选）
- SQLModel
- PostgreSQL + pgvector
- Alembic 迁移
- Pydantic Settings
- `pytest` / `FastAPI TestClient`
- `ruff` 代码质量

### 6.2 前端技术栈
- Node.js 24.15.0
- Vite 8.0.2
- Vue 3
- TypeScript
- Pinia
- Vue Router
- Tailwind CSS
- `Vitest`
- `ESLint`
- `Prettier`

### 6.3 开发与协作标准
- API 前缀统一使用 `/api`
- 本地后端地址为 `http://localhost:8000`
- 本地前端地址为 `http://localhost:5173`
- 前端通过 Vite proxy 转发至后端，避免跨域问题
- 所有真实密钥与凭证不提交仓库，使用 `.env.example` 作为模板
- 代码风格统一，前端遵循 ESLint/Prettier，后端遵循 ruff
- 所有重要变更通过 `README.md`、`backend/README.md`、`front/README.md` 进行说明

### 6.4 前后端交互规范
- 统一接口前缀：所有后端 API 均以 `/api` 开头，前端请求通过 `api/` 调用，以保证代理和路由一致性。
- 健康检查与基础数据：前端首轮加载调用 `/api/health`，用于验证后端可用性并展示基本状态。
- 数据格式约定：API 返回 JSON，状态字段统一使用 `status`、`data` 和 `message`，错误信息应在 `message` 中明确说明。
- 版本兼容：后端接口变更需增加版本说明或兼容层，避免前端因接口结构调整而直接失败。
- 前端请求处理：使用统一 API 客户端模块（例如 `front/src/api/client.ts`）封装 `fetch`/`axios`，在请求中处理公共错误、超时和基地址设置。
- 后端响应规范：确保后端接口返回 HTTP 状态码与业务结果一致，404/500 等错误使用标准状态码，业务失败使用 200 + 业务错误字段时需明确说明。
- 本地开发协作：前端开发时可以直接访问 `http://localhost:5173`，后端开发时通过 `http://localhost:8000` 提供 API；生产或集成环境下通过代理仅暴露前端入口。
- 联调流程：
  1. 后端先搭建健康检查接口和基础路由。
  2. 前端通过代理确认 `/api/health` 可达。
  3. 双方对齐接口文档与请求/响应示例后逐步增量开发。
  4. 接口变更时同步更新前端模块与后端文档。


## 8. 部署与环境
- 开发环境：本地运行，后端 8000，前端 5173
- 配置环境变量：后端 `backend/.env.example`，前端 `front/.env.example`
- 生产部署暂不包含完整平台脚本，后续可补充 Docker / CI/CD
