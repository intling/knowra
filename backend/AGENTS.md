# knowra 后端章程

## 1. 端侧定位

**knowra 后端** 是 knowra 的服务端与知识能力编排层，负责提供统一 API、管理数据与配置、承载文档处理、索引检索、模型服务适配、RAG 编排和后续知识图谱、评测能力的服务端边界。

后端不单独定义产品范围和业务优先级。产品目标、跨端边界和 OpenSpec 治理以根目录 `AGENTS.md` 为准；具体功能范围、验收标准和阶段任务以 OpenSpec 变更文档为准。

## 2. 技术栈基线

- **语言**：Python 3.14.5
- **Web 框架**：FastAPI
- **ASGI 服务**：Uvicorn，生产进程管理可使用 Gunicorn
- **ORM**：SQLModel
- **数据库**：PostgreSQL + pgvector
- **迁移**：Alembic
- **配置管理**：Pydantic Settings、`.env`、`.env.example`
- **测试**：pytest，必要时配合 `httpx` / FastAPI TestClient
- **格式化与静态检查**：ruff
- **包与命令管理**：uv

依赖版本以 `pyproject.toml` 和 `uv.lock` 为准。升级框架、数据库驱动、迁移工具或核心依赖时，应通过 OpenSpec 说明动机、影响范围和验证方式。

## 3. 后端职责

后端主要负责：

- FastAPI 应用入口、路由注册、中间件和生命周期管理。
- `/api` 前缀下的统一 HTTP API。
- 环境配置、日志、错误处理和跨域配置。
- PostgreSQL 数据访问、SQLModel 模型、Alembic 迁移和 pgvector 扩展。
- 文档解析、分块、元数据入库和索引构建的服务端流程。
- Embedding、重排、查询改写、HyDE、RAG 等模型相关能力的服务适配和编排。
- 检索服务、引用溯源、知识图谱和评测平台的后端扩展边界。

模型能力可以通过第三方平台、本地服务或后续可替换适配器接入。后端设计应关注接口抽象、配置隔离和可替换性，不应在章程层面预设唯一部署方式。

## 4. 推荐目录边界

后端代码应按 API、配置、数据、领域模型和服务逻辑分层。

```text
backend/app/
├── api/        # 路由、API 聚合与请求入口
├── core/       # 配置、日志、应用级基础设施
├── db/         # 数据库连接、session、初始化
├── models/     # SQLModel 数据模型
├── schemas/    # 请求/响应 schema
├── services/   # 业务服务、检索、文档处理、模型适配
└── main.py     # FastAPI 应用入口
```

路由层应保持薄层，只负责参数接收、依赖注入和响应返回；业务流程应下沉到 `services/` 或更明确的领域模块。

## 5. API 与配置规范

- 所有业务接口必须挂载在 `/api` 前缀下。
- 跨端接口的请求体、响应体和错误语义应保持稳定；破坏性变化需要通过 OpenSpec 说明。
- 敏感信息不得硬编码，统一通过 `.env`、`.env.example` 和配置模块管理。
- 第三方模型平台的 API Key、Base URL、模型名、超时和重试策略应配置化。
- 数据库连接串、CORS 来源、调试开关等环境差异必须通过配置隔离。
- 新增数据库表、索引或扩展时应配套 Alembic 迁移。

## 6. 服务设计规范

- 文档解析、分块、Embedding、检索、RAG、引用溯源等能力应通过清晰接口连接，避免互相写死。
- 服务层应优先表达业务流程，避免把第三方 SDK 调用散落在路由或模型中。
- 外部服务调用应封装在适配器或服务模块中，便于替换、测试和降级。
- 检索和生成链路应保留可观测信息，例如查询、召回片段、引用来源和错误原因。
- 数据模型和 API schema 应区分清楚，避免把数据库内部结构直接暴露为长期 API 契约。

## 7. 质量门禁

后端变更完成前，应根据影响范围运行必要验证命令：

```bash
uv run ruff check .
uv run ruff format --check .
uv run pytest
```

涉及数据库结构的改动，应验证 Alembic 迁移可执行。涉及 API 的改动，应至少覆盖状态码、响应结构和关键错误分支。涉及第三方模型服务的改动，应优先通过可替换配置、mock 或集成开关降低测试脆弱性。

## 8. 文档同步

以下变化需要同步更新文档：

- 启动方式、环境变量、端口、数据库或迁移流程变化。
- 后端目录结构或技术栈变化。
- API 路径、请求响应结构或错误语义变化。
- 文档处理、检索、模型适配、RAG 等核心服务边界变化。

同步目标通常包括 `backend/README.md`、根目录 `README.md`、相关 OpenSpec artifacts，以及必要时更新本章程。
